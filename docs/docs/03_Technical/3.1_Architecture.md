# 3.1 Architecture

## Overview

fc26-career-hub-api is a stateless **Fastify 4** HTTP service over **PostgreSQL** (Prisma) and **Redis** (ioredis). It owns persistence and business rules for FC Career Hub. There is no SSR and no client; the React SPA and OpenAI's Responses API are the consumers.

The codebase is **feature-based**: each domain capability is a folder under `src/features/` with a strict three-file split. This is the most important structural constraint to preserve.

## Process Layout

```
src/
├── app.ts        composition root — Fastify instance, plugins, swagger, route registration, error handler
├── server.ts     entrypoint — app.listen() locally; default-exports a Vercel handler
├── features/     domain capabilities (auth, saves, players, transfers, scout, chat, ...)
├── mcp/          MCP server (plugin, auth, server, tools/, resources/)
├── shared/
│   ├── lib/      prisma, redis, auth (Better Auth), fit-score-client
│   └── utils/    cache, errors, currency, auth-hooks, origins, fit-score-maps
└── types/        fastify.d.ts — augments FastifyRequest with `user` and `session`
```

### The three-file feature split

```
features/<name>/
├── <name>.routes.ts       Fastify schema (body/params/querystring) + handler wiring. No logic.
├── <name>.controller.ts   Extracts params/query/body/request.user, calls the service. No Prisma.
└── <name>.service.ts      Business logic, Prisma, cache, validation. The only layer that touches the DB.
```

Some features add helpers alongside these three: `scout-playbooks` has `scout-score.ts` + `scout-playbooks.types.ts`; `scouting` has `formations.ts`; `chat` has `persona.ts` + `chat.rate-limit.ts`; `__tests__/` folders hold Vitest suites.

## Application Bootstrap (`app.ts`)

In order:

1. **Fastify instance** — Pino logger (JSON to stdout in prod for Railway), `trustProxy: true`, AJV configured to allow an `example` keyword in schemas.
2. **CORS** — prod: callback validates `Origin` against the trusted-origins list (`shared/utils/origins.ts`, wildcard-aware); dev: `origin: true` (all). `credentials: true`.
3. **Compression** — gzip/br, threshold 1024 bytes.
4. **Content-type parser** — a custom `application/json` parser that treats an empty body as `{}` (so a `POST` with no payload doesn't 400).
5. **Swagger + Swagger UI** — OpenAPI doc with per-domain tags and a `bearerAuth` security scheme; UI served at `/docs`. `GET /` redirects to `/docs`.
6. **Public auth routes** — `authRoutes` registered at `/api` (no auth preHandler).
7. **MCP plugin** — `POST /mcp`, registered at the root; resolves its own auth from the bearer token.
8. **Protected scope** — a single `app.register(async (scope) => { scope.addHook('preHandler', requireAuth()); … })` block that registers **all** other feature routes at `/api`. This is the one place `requireAuth()` is wired.
9. **Schema error formatter** — rewrites AJV validation errors into a PT-BR `Campo inválido: '<field>' — <message>`.
10. **Global error handler** — see below.

## Request Lifecycle

```
HTTP request
  → CORS / compression / body parse
  → (protected scope) preHandler: requireAuth()
        → bearer token → Redis session:<token> (5 min) or Better Auth getSession
        → sets request.user, request.session  (else throws AppError 401)
  → controller (extract inputs)
  → service (validate → cache read → Prisma → cache write/invalidate)
  → reply
  → on throw: setErrorHandler maps it to a JSON envelope
```

`requireAuth()` (in `shared/utils/auth-hooks.ts`) proves authentication only. It does **not** check that `request.user.id` owns the `saveId` in the URL — that is each service's responsibility (and is currently inconsistent; see [3.7 Security](3.7_Security.md)).

## Error Handling

The global handler in `app.ts` produces two envelope shapes:

```jsonc
{ "error": "Save não encontrado.", "statusCode": 404 }                       // AppError without code
{ "error": "SHIRT_NUMBER_CONFLICT", "message": "...", "statusCode": 409 }    // AppError with code
```

Mapping rules:
- `AppError` → its `statusCode`; if `.code` is set, the code goes in `error` and the human text in `message`.
- Prisma `P2025` → 404, `P2003` → 400, `P2002` → 409.
- `error.validation` (AJV) → 400 with the formatted message.
- Any other `statusCode < 500` → passthrough.
- Otherwise → 500 `Erro interno do servidor.` (full error logged via Pino + `console.error('[500]', …)`).

## Authentication & Session Resolution

- **Better Auth** (`shared/lib/auth.ts`): email/password (no email verification), `admin` + `bearer` plugins, `plan` as a read-only additional field, global rate limiter (60s/100, toggled by `DISABLE_RATE_LIMIT`), prod cookie attributes (`sameSite: none`, `secure`, `partitioned`).
- **Session cache**: `requireAuth()` reads `session:<token>` from Redis (5 min TTL) before falling back to `auth.api.getSession()`. Sign-out calls `invalidateSessionCache(token)`.
- **`requireRole` / `requirePlan`**: implemented (with `PLAN_HIERARCHY`, admin bypass) but **not attached to any route**.

## MCP Subsystem

`POST /mcp` is registered **outside** the `requireAuth()` scope because it authenticates itself (`mcp/auth.ts` → `mcp:session:<token>` cached 5 min). Per request it: resolves the user, applies an MCP rate limit, instantiates a fresh `McpServer` (`mcp/server.ts`) bound to the user context, and pipes the request through a `StreamableHTTPServerTransport`. Tools (`mcp/tools/`) and resources (`mcp/resources/`) are all **read-only** and query strictly under `ctx.userId`. The chat module is the only caller path (via OpenAI). See [Module 3.6.11](Modules/3.6.11_MCP_Server.md).

## Integration Topology

```
React SPA ──bearer──> /api/*  (Fastify)
                         │
                         ├── Prisma ──> PostgreSQL (Neon; pooled, DIRECT_URL for migrations)
                         ├── ioredis ─> Redis (sessions, read caches, rate-limit counters)
                         ├── chat ────> OpenAI Responses API ──MCP(bearer)──> /mcp (same service)
                         └── fc26 ────> Fit-score service (HTTP, 3s timeout, fail-open)
```

## Architectural Boundaries

### Boundary 1 — routes vs. controllers vs. services
Routes only declare schema; controllers only marshal request/response; services own all logic and I/O. A Prisma call in a controller, or business logic in a route, is a layering violation.

### Boundary 2 — services own their cache and invalidation
Each service defines its `TTL` const and its key scheme, and is responsible for invalidating on writes. There is no central cache registry; cache failures never propagate.

### Boundary 3 — shared singletons
`prisma` and `redis` are module-level singletons in `shared/lib`. Features import them; they never construct their own. `auth` (Better Auth) is likewise a single instance.

### Boundary 4 — stateless process
No per-process mutable state survives a request beyond the singletons and Redis. This is what makes horizontal scaling on Railway safe (see [3.8 Scaling and Performance](3.8_Scaling_and_Performance.md)).

## Deployment Shape

- **Railway** (`railway.json`): build `npm run build` (tsc → `dist/`), pre-deploy `prisma migrate deploy`, start `npm start` (`node dist/server.js`), health check `GET /`, single replica.
- **Docker** (`Dockerfile`): multi-stage (deps → builder → runtime on `node:20-alpine`), `openssl` for Prisma. `docker-compose.yml` runs api + Postgres + Redis (+ InfluxDB/Grafana for load tests) locally.
- **Vercel** (`vercel.json`): legacy serverless config; `server.ts` exports a handler guarded by `process.env.VERCEL`. Prod is Railway.
