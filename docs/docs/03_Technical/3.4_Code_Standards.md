# 3.4 Code Standards

## Base

- **TypeScript**, ESM (`"type": "module"`). Local imports use explicit `.js` specifiers (e.g., `import { prisma } from '../../shared/lib/prisma.js'`) even though the source is `.ts` — this is required for Node ESM after `tsc`.
- **Node 20**. No bundler; `tsc` emits `dist/`, run with `node dist/server.js`. Dev uses `tsx watch`.
- **Vitest** for tests. No linter config is committed; keep the existing style.
- No path aliases — imports are relative. Keep them shallow by respecting the feature boundary.

## Folder Conventions

### Feature = three files
```
features/<name>/
├── <name>.routes.ts       Fastify schema + handler wiring
├── <name>.controller.ts   request/response marshalling
├── <name>.service.ts      business logic + Prisma + cache
├── <name>.types.ts        (optional) shared types for the feature
├── <helpers>.ts           (optional) pure helpers (scoring, formations, persona)
└── __tests__/             (optional) Vitest suites
```

Keep the boundaries crisp:
- **routes**: declare `schema` (tags/summary/body/params/querystring/response). No logic, no Prisma.
- **controller**: read `request.params/query/body/user`, call the service, shape the reply (status code). No Prisma, no validation beyond what the schema did.
- **service**: everything else — domain validation, Prisma, cache, transactions, formatting.

### `shared/`
- `lib/` — stateful singletons and external clients (`prisma`, `redis`, `auth`, `fit-score-client`).
- `utils/` — pure-ish helpers (`cache`, `errors`, `currency`, `auth-hooks`, `origins`, `fit-score-maps`).

### `mcp/`
- `plugin.ts` (route), `auth.ts` (self-auth), `server.ts` (factory), `tools/`, `resources/`, `schemas.ts`, `context.ts`, `utils.ts`.

## Naming

- **Files**: kebab-case for features (`scout-playbooks.service.ts`), kebab-case in `shared` (`auth-hooks.ts`, `fit-score-client.ts`).
- **Functions**: camelCase verbs (`createSave`, `listFc26Players`, `evaluateSigningFit`).
- **Types/interfaces**: PascalCase. API/dataset types may carry a domain prefix (`Fc26PlayerWithFitScore`, `ScoutScoreResult`).
- **Cache keys**: colon-namespaced (`save:<id>:players:active:<season>`).
- **Enums**: match the Prisma casing exactly (`Position`, `PlayerStatus`, `TransferType`, `CupResult`).

## Request Validation

- Validation lives in the Fastify `schema` as JSON Schema (AJV). Prefer `additionalProperties: false` on bodies to reject unknown fields (players/team-stats/playbooks do this).
- Enumerate allowed values with `enum` (positions, statuses, transfer types, cup results, foot, formation).
- Use `pattern` for formats (season `^\d{4}\/\d{2}$`), and `minimum`/`maximum` for ranges (age 15–45, ovr 40–99, shirt 1–99).
- The schema error formatter turns AJV errors into `Campo inválido: '<field>' — <message>`.
- Domain validation that JSON Schema can't express (shirt-number uniqueness, alternative-position rules, season-string sanity, ownership) lives in the **service**.

## Error Handling

- Throw `AppError(message, statusCode, code?)` / `NotFoundError(message)` from services. Never `reply.status(500)` by hand for expected errors — let the global handler shape the envelope.
- Set `AppError.code` only when the client needs to branch (e.g., `SHIRT_NUMBER_CONFLICT`). Otherwise the message is the contract.
- Catch a Prisma error only to translate it locally (e.g., `P2025` → a friendlier `NotFoundError` in `updatePlayerStats`); otherwise let the global handler map P2025/P2003/P2002.

## Caching

- One `TTL` const object per service. Pattern: `cacheGet` → on miss query → `cacheSet`.
- Cache reads only; never cache a write result that another path could invalidate inconsistently.
- Invalidate **every** affected key on a write. Use point invalidation by default; reserve `cacheInvalidatePattern` for cascades.
- Cache calls must not be in the critical path's failure mode — they already swallow errors; don't `await` them inside a transaction.

## Transactions

- Wrap any multi-row invariant in `prisma.$transaction(async (tx) => …)`. Established transactional flows: save creation, season advance, club change, transfer creation/update, default-playbook flip.
- Read the entities you need **before** the transaction when they don't participate in the invariant (e.g., `createTransfer` loads `save` once and reuses it), to keep transactions short.

## Currency

- Two units (see [3.2](3.2_Data_Model.md)). Always format via `shared/utils/currency.ts`; never inline. Services return both raw and `*Formatted`.

## Ownership & Auth (standard to follow)

- Save-scoped services take `userId` (from `request.user.id` in the controller) and assert ownership:
  ```ts
  const save = await prisma.save.findFirst({ where: { id: saveId, userId }, select: { id: true } })
  if (!save) throw new NotFoundError('Save não encontrado.')
  ```
  This is the `assertSaveAccess` pattern in `scout-playbooks`/`shortlist`/`saved-searches`. **Apply it to every save-scoped operation.**

## MCP Tools

- Register tools in `mcp/tools/<area>.ts` and wire them in `mcp/tools/index.ts`. Each tool: a Zod `inputSchema`, a clear `description` (the model reads it to decide when to call), and a handler that resolves the save via `resolveSaveId(ctx.userId, saveId)` and queries strictly under `ctx.userId`.
- Tools return Markdown text in `content[0].text`. Keep tools **read-only**.

## Comments & Language

- Comments are sparse and explain **why** (a constraint, a workaround), not what. Existing inline comments are mostly PT-BR; keep new ones short and in the surrounding style.
- Swagger `summary`/`description` strings are PT-BR. Error messages are PT-BR. Code identifiers and these docs are English.

## Testing

- Vitest. Suites live in `__tests__/` next to the code (`clubs`, `scout-score`, `scouting`, `mcp`) or under `shared/.../__tests__` (`currency`, `errors`, `fit-score-maps`, `origins`, `fit-score-client`).
- Run all: `npm test`. Single file: `npx vitest run src/features/clubs/__tests__/clubs.service.test.ts`.
- Priorities to cover: ownership checks, save-creation cascade, season advance, transfer balance math, shirt-number conflict.

## Conventions to preserve

- The three-file feature split; helpers as extra files, never logic in routes/controllers.
- `.js` import specifiers on local modules.
- `TTL` const per service; colon-namespaced cache keys.
- `AppError`/`NotFoundError` for expected failures; `code` only when client-branchable.
- PT-BR for user-facing strings; English for code and docs.
