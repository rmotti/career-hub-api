# 3.6.9 Scout (Playbooks, Scoring, Shortlist, Saved Searches, Scouting)

## Responsibility

The recruitment surface built on top of the [FC26 Dataset](3.6.8_FC26_Dataset.md). Four cooperating features:

- **Scout Playbooks** — configurable weighting profiles + the scoring engine.
- **Scouting** — squad-gap detection, transfer-target search, per-signing evaluation.
- **Shortlist** — per-save list of tracked dataset players.
- **Saved Searches** — per-save persisted filter sets.

This is the paid (PRO) surface, gated by plan via `requirePlan('PRO')`.

## Scoring Engine (`scout-score.ts`)

- **Components** (`weights`): `overall, age, historicalFit, potential, marketValue, wage`. Default playbook ("Equilibrado"): overall 30 / potential 15 / age 15 / historicalFit 15 / marketValue 25 (no `wage`).
- **Preferences**: `objective` (`balanced|title|youth|rebuild` — now feeds **only** the fit-score model), `maxMarketValue` (millions), `maxWage` (thousands). `idealAgeMin/Max` were **removed** (the age curve replaced the ideal band; the route schema still accepts them but ignores them, for FE back-compat).
- Each component is scored 0–100 (**B-003**): **overall** maps `OVR 50→0 / 95→100`; **potential** is upside `(potential − ovr) / 20`; **age** is a convex curve (100 @16, accelerating per-year penalty k = 2/3/5/7); **historicalFit** is `fit × 100` or **0** when no profile (kept, not dropped); **marketValue**/**wage** are budget headroom `100 × (1 − value/B)`, `B = cap ?? save.budget` (marketValue) / `maxWage` only (wage, opt-in) — a player **over the explicit cap is excluded** from the list. The final `scoutScore` is the **weight-normalized average of available components**; `scoutScoreConfidence` derives from historical-fit availability; a per-component `scoutScoreBreakdown` is returned.
- `/scout/evaluate` resolves the playbook (explicit `playbookId` → inline `playbook` → the save's default → the built-in default), reads `save.budget` for the cost gradient, runs `listFc26Players` (with fit-score + the cap as a hard filter), scores each player, and sorts by `scoutScore desc`.

## Scouting Analysis (`scouting.service.ts`)

- **`identifyGaps`** — compares the active squad to a formation's per-position `ideal`/`min` (`formations.ts`), grading each gap `critical` (below min) / `moderate` (below ideal, or avg age ≥ 31) / `low` (best OVR < 75), sorted by severity.
- **`searchTransferTargets`** — thin wrapper over `listFc26Players` (position, maxAge, minOverall, maxValue, optional fit-score).
- **`evaluateSigningFit`** — compares a `sofifaId`'s cost vs. `budget` and OVR vs. the best same-position squad player → verdict `strong|reasonable|poor`. The `alternatives` field is reserved for a future suggestion list.

## Local Domain Rules

1. One default playbook per save (`@@unique([saveId, name])`; setting default flips others in a transaction); needs ≥1 positive weight.
2. Shortlist unique per `(save, fc26Player)`; notes ≤ 500; priority `LOW|MEDIUM|HIGH`.
3. Saved searches unique per `(save, name)`; name ≤ 80.
4. Ownership is confirmed here: `scout-playbooks`/`shortlist`/`saved-searches` use `assertSaveAccess(saveId, userId)`; `scouting` uses `getActiveStint` (`findFirst { id, userId }`).
5. This is the paid (PRO) surface; plan gating is applied via `requirePlan('PRO')`.

## Fit-score club-name aliases (sync runbook)

App/FC26 club names don't always match the fit-score-svc club vocabulary, so `toFitScoreClubName` (`shared/utils/fit-score-maps.ts`) consults `MANUAL_CLUB_NAME_BY_LEAGUE_MAP` from the **generated** `shared/utils/fit-score-club-aliases.generated.ts`. That file is committed; its source worksheet CSV (`data/…Worksheet.csv`) is **gitignored**, and regeneration needs the sibling `fit-score-svc` Python venv + `club_profiles.pkl`. Forgetting to re-sync after club names change silently degrades scouting (a wrong/absent alias → fit-score falls back and the player scores against the wrong club profile).

**When to re-sync:** a new FC season's club renames, or when fit-score-svc updates its club vocabulary (`club_profiles.pkl`).

**How (local — not CI):**
```bash
# prereqs: sibling repo ../fit-score-svc with its venv + artifacts/club_profiles.pkl
#   override with FIT_SCORE_PYTHON / FIT_SCORE_ARTIFACTS_DIR if they live elsewhere
npm run fit-score:clubs:sync           # rewrites the .generated.ts (validates every alias against club_profiles.pkl)
git add src/shared/utils/fit-score-club-aliases.generated.ts
```

**Staleness guard:** the sync stamps the generated file with `// worksheet-sha256: <hash>`. `npm run fit-score:clubs:check` recomputes the worksheet hash and **fails (exit 1)** if it drifted — the "edited the CSV but forgot to re-sync" reminder. It **skips** (exit 0) when the worksheet isn't present locally (e.g. CI / a teammate without the CSV), so it never blocks them. Runtime computation isn't viable — the club vocabulary only exists in the offline `club_profiles.pkl`, not behind an API.
