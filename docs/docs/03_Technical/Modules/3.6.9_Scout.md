# 3.6.9 Scout (Playbooks, Scoring, Shortlist, Saved Searches, Scouting)

## Responsibility

The recruitment surface built on top of the [FC26 Dataset](3.6.8_FC26_Dataset.md). Four cooperating features:

- **Scout Playbooks** — configurable weighting profiles + the scoring engine.
- **Scouting** — squad-gap detection, transfer-target search, per-signing evaluation.
- **Shortlist** — per-save list of tracked dataset players.
- **Saved Searches** — per-save persisted filter sets.

This is the "PRO" surface — gated in the frontend, **not** in the API (see [3.7 Security #2](../3.7_Security.md#2-no-server-side-plan-gating)).

## Scoring Engine (`scout-score.ts`)

- **Components** (`weights`): `overall, age, historicalFit, potential, marketValue, wage`. Default playbook ("Equilibrado"): overall 35 / age 20 / historicalFit 25 / potential 20.
- **Preferences**: `objective` (`balanced|title|youth|rebuild`, each with a default ideal age range), `idealAgeMin/Max` (clamped 15–45), `maxMarketValue`, `maxWage`.
- Each enabled component is scored 0–100 (overall/potential clamp the raw value; age scores 100 inside the ideal band and decays 8/yr outside; market/wage score against the cap or a log decay; `historicalFit` uses the fit-score × 100, or is marked unavailable). The final `scoutScore` is the **weight-normalized average of available components**; `scoutScoreConfidence` derives from historical-fit availability; a per-component `scoutScoreBreakdown` is returned.
- `/scout/evaluate` resolves the playbook (explicit `playbookId` → inline `playbook` → the save's default → the built-in default), runs `listFc26Players` (with fit-score), scores each player, and sorts by `scoutScore desc`.

## Scouting Analysis (`scouting.service.ts`)

- **`identifyGaps`** — compares the active squad to a formation's per-position `ideal`/`min` (`formations.ts`), grading each gap `critical` (below min) / `moderate` (below ideal, or avg age ≥ 31) / `low` (best OVR < 75), sorted by severity.
- **`searchTransferTargets`** — thin wrapper over `listFc26Players` (position, maxAge, minOverall, maxValue, optional fit-score).
- **`evaluateSigningFit`** — compares a `sofifaId`'s cost vs. `budget` and OVR vs. the best same-position squad player → verdict `strong|reasonable|poor`. `alternatives` is currently always `[]` (see [Tech Debt #9](../3.5_Technical_Debt.md)).

## Local Domain Rules

1. One default playbook per save (`@@unique([saveId, name])`; setting default flips others in a transaction); needs ≥1 positive weight.
2. Shortlist unique per `(save, fc26Player)`; notes ≤ 500; priority `LOW|MEDIUM|HIGH`.
3. Saved searches unique per `(save, name)`; name ≤ 80.
4. **Ownership IS enforced** here: `scout-playbooks`/`shortlist`/`saved-searches` use `assertSaveAccess(saveId, userId)`; `scouting` uses `getActiveStint` (`findFirst { id, userId }`). This is the pattern the core CRUD modules should adopt.
5. Plan gating is **not** enforced server-side — gated only in the SPA.
