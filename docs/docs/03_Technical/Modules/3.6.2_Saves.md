# 3.6.2 Saves and Careers

## Responsibility

A "save" is one Career Mode career and the root context for almost everything else. This module owns listing, creation (with the initial cascade), updates (including the irreversible season-advance cascade), and deletion.

## Files

```
features/saves/
├── saves.routes.ts        schema + wiring
├── saves.controller.ts    pulls userId from request.user, calls the service
├── saves.service.ts       listSaves, getSaveById, createSave, updateSave, deleteSave, listDeletedSaves, restoreSave
└── snapshots.service.ts   createSnapshot, restoreSnapshot, listSnapshots, createManualSnapshot, writeAudit
```

## Endpoints

| Method | Path | Notes |
|---|---|---|
| GET | `/api/saves` | List (excludes archived) with `currentClubStint` + formatted budget/balance. |
| GET | `/api/saves/deleted` | List archived (soft-deleted) saves — the "trash". |
| GET | `/api/saves/:saveId` | Detail + `clubStints` + `availableSeasons`. **404 if not owner or archived.** |
| POST | `/api/saves` | `{ name, club, budget, europeanCompetitionId? }`. |
| PATCH | `/api/saves/:saveId` | Update; `currentSeason` triggers the cascade (auto-snapshots first). |
| DELETE | `/api/saves/:saveId` | **Soft-delete by default** (archives, reversible, auto-snapshots). `?purge=true` = permanent. Requires `?confirm=<saveId>`. |
| POST | `/api/saves/:saveId/restore` | Un-archive a soft-deleted save. |
| GET | `/api/saves/:saveId/audit` | Activity log: irreversible/recovery mutations, newest first (max 100). |
| GET | `/api/saves/:saveId/snapshots` | List restore points (`reason`, `createdAt`). |
| POST | `/api/saves/:saveId/snapshots` | Create a manual save-point. |
| POST | `/api/saves/:saveId/snapshots/:snapshotId/restore` | Revert the whole save tree to a snapshot. |

## Data Flow

### Creation (`createSave`)
1. Validate `club` against the in-memory list (`clubExists`).
2. Resolve the club's league → country → competition ids (+ optional European competition).
3. In one transaction: create `Save` (`currentYear 2025`, `currentSeason '2025/26'`, `balance = budget`), the initial `ClubStint` (`isCurrent`, `startYear '2025'`), and one `TeamSeasonStats` per competition.
4. Invalidate `user:<userId>:saves`.

### Read
- `listSaves` → cache `user:<userId>:saves` (15 min); each save carries `currentClubStint` and `*Formatted` money.
- `getSaveById` → cache `save:<saveId>` (30 min); enforces ownership (`save.userId !== userId → 404`) and computes `availableSeasons` from the current stint's team stats.

### Season advance (`updateSave` with a new `currentSeason`)
Runs one transaction (see [2.1 Business Rules #11](../../02_Domain/2.1_Business_Rules.md)):
- **Take a `pre-season-advance` snapshot first** (in the same transaction → atomic with the advance) and write a `save.season_advance` audit row.
- Upsert trophies for ending-season wins (league pos 1 / cup `Campeao`).
- Snapshot `PlayerOvrHistory` for active players.
- Age active players under 45 by +1.
- Create `TeamSeasonStats` (per competition) and `PlayerSeasonStats` (per active player) for the new season.
- Re-attach players loaned out from this stint (`status Loan` → `Role`, fresh stats).
- If `budget` is sent, reset `balance = budget`.
Then `cacheInvalidatePattern('save:<id>:*')` + point-invalidate the save and the saves list.

### Deletion & recovery (`deleteSave`, `restoreSave`)
`deleteSave` checks ownership and **requires `confirm === saveId`** (else `400 DELETE_CONFIRMATION_REQUIRED`):
- **Default (soft):** take a `pre-delete` snapshot + `save.soft_delete` audit, then set `deletedAt`. Reversible. List/detail reads filter `deletedAt: null`, so it vanishes from the UI but the data stays.
- **`?purge=true` (hard):** write a `save.purge` audit (the `AuditLog` has no FK to `Save`, so it survives), then `delete` (`onDelete: Cascade` removes all children **and** the save's snapshots). Terminal.

`restoreSave` clears `deletedAt` (un-archive) for the "deleted by accident" case — the soft-delete didn't touch the data, so this alone brings the save back as-is.

### Snapshots & undo (`snapshots.service.ts`)
A `SaveSnapshot` is a JSON dump of the whole save tree (`clubStints`, `players`, `transfers`, all season stats, trophies, OVR history, playbooks, shortlist, saved searches) with an IDs-preserved payload. Created automatically before the two irreversible flows (season advance, soft-delete) and manually via the snapshots endpoint. Kept to the latest **10 per save** (older auto-pruned).

`restoreSnapshot` runs one transaction: delete the current children (reverse-FK order) → re-insert from the payload (FK order, original IDs) → reset the `Save` scalars and clear `deletedAt`. This is the **undo** for a season advance (or any reverted state). `Competition`/`Fc26Player` are external tables and never touched.

### Audit trail (`AuditLog`)
Every irreversible/recovery mutation writes an `AuditLog` row — `save.season_advance`, `save.soft_delete`, `save.purge`, `save.restore`, `save.snapshot_create`, `save.snapshot_restore`, `save.finance_edit` (direct budget/balance edits, with `{ from, to }`), plus the cross-module `transfer.reverse`, `player.release`, `clubstint.change` and `squad.import`. No FK to `Save` → audit history outlives a purge. Exposed read-only via `GET /saves/:saveId/audit`. The `SaveSnapshot`/`writeAudit` helpers are reused by the transfers, players and club-stints modules (see [3.6.5](3.6.5_Transfers.md), [3.6.4](3.6.4_Players.md), [3.6.3](3.6.3_Club_Stints.md)).

## Local Domain Rules

1. Owned by exactly one user; reads/writes here are scoped by `userId` (this module **does** enforce ownership — the model the others should follow).
2. Exactly one `ClubStint.isCurrent` per save.
3. `budget`/`balance` are millions of €; `balance` starts equal to `budget`.
4. `currentSeason` is `"YYYY/YY"`; never set directly — it goes through `updateSave`. The advance is destructive but now **recoverable** via the auto `pre-season-advance` snapshot (restore reverts the whole tree).
5. New saves start in **2025/26** (the create logic sets this explicitly, ahead of the schema's column defaults).
6. Delete is **soft by default** and gated by `?confirm=<saveId>`; only `?purge=true` is permanent. Reads filter `deletedAt: null`.
7. Snapshots/audit are scoped by save ownership; snapshots cascade away on purge, audit rows do not.

## Extension Points

- ~~Save export/snapshot (JSON) before the irreversible season advance~~ ✅ done (`SaveSnapshot`).
- ~~An "undo last season" path~~ ✅ done (`restoreSnapshot` from the `pre-season-advance` snapshot).
- Save duplication (clone a career) — `collectSavePayload` could be reused to fork a save under a new id.
- Download a snapshot payload as a JSON file (true export) / import.
- Aggregated career analytics built on `PlayerOvrHistory` + season stats.
- A scheduled job to purge saves archived for > N days.

## Auto-audit

- Cascade and ownership behavior read from `saves.service.ts` (`createSave`, `updateSave`, `getSaveById`, `deleteSave`, `restoreSave`).
- Snapshot/restore/audit behavior read from `snapshots.service.ts`; soft-delete column + `SaveSnapshot`/`AuditLog` models from `schema.prisma`.
- 2025/26 start confirmed in `createSave` vs. the `Save` model defaults.
- Cache keys/TTLs read from the service's `TTL` const.
