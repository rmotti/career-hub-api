# 3.6.1 Auth and Sessions

## Responsibility

Owns account creation, sign-in, session resolution, and the per-request auth guard. Wraps **Better Auth** and exposes a thin set of public endpoints plus the `requireAuth()`/`requirePlan()`/`requireRole()` hooks the rest of the API depends on.

## Files

```
features/auth/
└── auth.routes.ts          public /api/auth/* endpoints (proxy to Better Auth)
shared/lib/auth.ts          Better Auth instance (email/password, admin + bearer plugins)
shared/utils/auth-hooks.ts  requireAuth / requireRole / requirePlan + session cache
shared/utils/origins.ts     trusted-origins resolution (CORS + Better Auth)
types/fastify.d.ts          augments FastifyRequest with `user` and `session`
```

There is no `auth.controller.ts`/`auth.service.ts` — the routes hand the raw request to `auth.handler()`.

## Endpoints (public)

| Method | Path | Notes |
|---|---|---|
| POST | `/api/auth/sign-up/email` | `{ name, email, password }` → `{ token, user }`. No email verification. |
| POST | `/api/auth/sign-in/email` | `{ email, password }` → `{ token, user }`. |
| GET | `/api/auth/session` | `{ session, user }` or `null` (Better Auth's 404 is mapped to `200 null`). |
| POST | `/api/auth/sign-out` | `{ success: true }`; invalidates `session:<token>`. |
| GET/POST | `/api/auth/*` | Hidden catch-all → Better Auth internals (reset, etc.). |

`toBetterAuthRequest()` adapts the Fastify request into a WHATWG `Request` for `auth.handler()`, then copies status/headers/body back onto the reply.

## Better Auth Configuration

- **email/password**, `requireEmailVerification: false`.
- Plugins: `admin` (`defaultRole: 'user'`, `adminUserIds: []` — set via env if needed) and `bearer` (opaque token auth).
- `user.additionalFields.plan` — `UserPlan` string, `input: false` (clients can't set it at sign-up).
- `trustedOrigins` from `getTrustedOrigins()`.
- `rateLimit`: window 60s, max 100, `enabled` unless `DISABLE_RATE_LIMIT=true`.
- Prod cookie attributes: `sameSite: 'none'`, `secure: true`, `partitioned: true`.

## Session Resolution (the hot path)

`requireAuth()` runs as the single `preHandler` on the protected scope:

1. Extract the bearer token.
2. `cacheGet('session:<token>')` — Redis, 5 min TTL.
3. On miss → `auth.api.getSession({ headers, query })`, then `cacheSet`.
4. Set `request.user` / `request.session`, or throw `AppError('Não autenticado.', 401)`.

Sign-out calls `invalidateSessionCache(token)` so the cached session can't outlive an explicit logout. Other revocations (ban, plan change) lag by up to the 5-min TTL — see [3.7 Security](../3.7_Security.md#6-session-cache-revocation-lag).

## Guards (`auth-hooks.ts`)

| Guard | Effect | Used? |
|---|---|---|
| `requireAuth()` | 401 if no valid session; sets `request.user` | ✅ on the whole protected scope |
| `requireRole('admin')` | 403 unless `user.role === role` | ❌ not wired to any route |
| `requirePlan('PRO')` | 403 unless `PLAN_HIERARCHY[plan] ≥` target; **admin bypasses** | ❌ not wired to any route |

`PLAN_HIERARCHY = { FREE: 0, PRO: 1, PREMIUM: 2 }`.

## Local Domain Rules

1. A bearer token is an opaque session id — treat it as a secret; it grants full account access until expiry.
2. Sign-up cannot self-assign a plan (`input: false`).
3. `GET /auth/session` returns `null` (200) when there's no session, not 404 — the route normalizes Better Auth's response.
4. The session cache is keyed by raw token; never log the `authorization` header (see [Security #8](../3.7_Security.md#8-errorlog-hygiene--low)).

## Extension Points

- Wire `requirePlan('PRO')` onto the Scout scope to make the paywall real (see [3.7 Security](../3.7_Security.md#2-no-server-side-plan-gating)).
- Add OAuth/social providers via Better Auth plugins.
- Add email verification + password-reset UX (the catch-all already proxies the endpoints).
- Mint a short-lived, MCP-scoped token to avoid sending the full session token to OpenAI ([Security #3](../3.7_Security.md#3-user-session-token-transits-openai)).
