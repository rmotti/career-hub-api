# 3.6.1 Auth and Sessions

## Responsibility

Owns account creation, sign-in, session resolution, and the per-request auth guard. Wraps **Better Auth** and exposes a thin set of public endpoints plus the `requireAuth()`/`requirePlan()`/`requireRole()` hooks the rest of the API depends on.

## Files

```
features/auth/
└── auth.routes.ts          public /api/auth/* endpoints (proxy to Better Auth) + cookie/CSRF emission
shared/lib/auth.ts          Better Auth instance (email/password, admin + bearer plugins)
shared/utils/auth-hooks.ts  requireAuth / requireRole / requirePlan + csrfProtection + session cache
shared/utils/cookies.ts     session_token / csrf_token (de)serialization, CSRF gen, timing-safe compare
shared/utils/origins.ts     trusted-origins resolution (isTrustedOrigin + isCredentialedOriginAllowed)
types/fastify.d.ts          augments FastifyRequest with `user` and `session`
```

There is no `auth.controller.ts`/`auth.service.ts` — the routes hand the raw request to `auth.handler()`.

## Endpoints (public)

| Method | Path | Notes |
|---|---|---|
| POST | `/api/auth/sign-up/email` | `{ name, email, password }` → `{ csrfToken, user }`. Sets `session_token` (httpOnly) + `csrf_token` cookies. The session token is **no longer returned in the body** (post-cutover) — it lives only in the httpOnly cookie. No email verification. |
| POST | `/api/auth/sign-in/email` | `{ email, password }` → `{ csrfToken, user }`. Sets `session_token` + `csrf_token` cookies; token not in body. |
| GET | `/api/auth/csrf` | `{ csrfToken }`. Returns the current `csrf_token` cookie value or mints one. The SPA calls it on boot to re-sync the token (cross-site, it can't read the cookie). |
| GET | `/api/auth/session` | `{ session, user }` when authenticated, else **`401`**. Resolves the session via the shared `getSession()` (cookie-first → Bearer injection), **not** by forwarding the raw request to Better Auth — that path can't read our `session_token` cookie cross-site. |
| POST | `/api/auth/sign-out` | `{ success: true }`; invalidates `session:<token>` and **clears** both cookies (`Max-Age=0`). |
| GET/POST | `/api/auth/*` | Hidden catch-all → Better Auth internals (reset, etc.). |

`toBetterAuthRequest()` adapts the Fastify request into a WHATWG `Request` for `auth.handler()`. `forwardHeaders()` then copies status/headers back (handling multi-value `set-cookie` via `getSetCookie()`, dropping `content-length`/`content-encoding`) and appends our cookies. `handleSessionResponse()` wraps sign-in/sign-up: on success it parses the body, uses the `token` to set the `session_token` cookie and then **deletes it from the body**, emits `csrf_token`, and adds `csrfToken` to the body + `X-CSRF-Token` header.

## Better Auth Configuration

- **email/password**, `requireEmailVerification: false`.
- Plugins: `admin` (`defaultRole: 'user'`, `adminUserIds: []` — set via env if needed) and `bearer` (opaque token auth).
- `user.additionalFields.plan` — `UserPlan` string, `input: false` (clients can't set it at sign-up).
- `trustedOrigins` from `getTrustedOrigins()`.
- `rateLimit`: window 60s, max 100, `enabled` unless `DISABLE_RATE_LIMIT=true`.
- Prod cookie attributes: `sameSite: 'none'`, `secure: true`, `partitioned: true`.

**CORS (`app.ts`)** — credentialed: `credentials: true`; prod origin is **exact-match only** via `isCredentialedOriginAllowed` (no `*`, no wildcard — wildcards on `*.vercel.app` are forgeable). `allowedHeaders: Content-Type, Authorization, X-CSRF-Token`; `exposedHeaders: X-CSRF-Token, set-auth-token`; preflight cached 24h. Dev reflects any origin (`NODE_ENV !== 'production'`).

## Session Resolution (the hot path)

The protected scope runs `csrfProtection()` as an **`onRequest`** hook (earliest phase — before body validation, so a forged write returns 403 instead of being masked by a 400 on an invalid body) and `requireAuth()` as a `preHandler`.

`extractSessionToken(request)` decides the token source:

1. **Cookie first** — the httpOnly `session_token` cookie (new, XSS-resistant flow).
2. **Bearer fallback** — `Authorization: Bearer <token>` (legacy clients; kept for a zero-downtime transition).

Both carry the **same value** (the raw `token` returned at login), so the cache key and validation converge. `requireAuth()` then:

1. `extractSessionToken()` → token.
2. `cacheGet('session:<token>')` — Redis, 5 min TTL.
3. On miss → `auth.api.getSession({ headers: { authorization: 'Bearer <token>' }, query })` — the token is **always** injected as a Bearer header so Better Auth's `bearer` plugin validates the cookie and header paths identically (it re-signs an unsigned token with the secret). Then `cacheSet`.
4. Set `request.user` / `request.session`, or throw `AppError('Não autenticado.', 401)`.

Sign-out calls `invalidateSessionCache(token)` (token from cookie **or** bearer) so the cached session can't outlive an explicit logout. The 5-min TTL is the freshness window for session state.

## Cookie auth + CSRF (cross-site hardening)

Frontend (Vercel) and API (Railway) are **different sites**, so the session cookie must be `SameSite=None; Secure` and CORS must allow credentials. `shared/utils/cookies.ts` centralizes this.

**Cookies emitted** (prod attributes; dev falls back to `SameSite=Lax` without `Secure` so localhost works):

| Cookie | httpOnly | Attributes | Purpose |
|---|---|---|---|
| `session_token` | ✅ | `Path=/; Max-Age=604800; Secure; SameSite=None; Partitioned` | Session token, unreadable by JS (XSS can't exfiltrate it). |
| `csrf_token` | ❌ | `Path=/; Max-Age=604800; Secure; SameSite=None; Partitioned` | Double-submit CSRF token. |

`Max-Age` matches Better Auth's default session `expiresIn` (7 days). No `Domain` → host-only. `Partitioned` (CHIPS) mirrors Better Auth's prod cookie attributes and future-proofs against third-party-cookie blocking.

**CSRF (double-submit), `csrfProtection()` (runs `onRequest`):** because the session cookie is `SameSite=None`, the browser sends it cross-site → writes need proof of origin. The hook enforces, **only on cookie-authenticated writes** (`POST/PUT/PATCH/DELETE`), that the `X-CSRF-Token` header equals the `csrf_token` cookie (timing-safe compare). It **skips**:
- safe methods (`GET/HEAD/OPTIONS`);
- requests with no `session_token` cookie — i.e. Bearer or unauthenticated clients. Bearer isn't auto-attached by the browser, so it's not CSRF-able; this is what keeps legacy clients working during the transition.

The custom header also forces a CORS preflight, which the exact-origin allowlist gates — a second barrier. **Cross-site caveat:** the SPA cannot read the API's `csrf_token` cookie via `document.cookie` (different domain), so it obtains the value from the login body / `X-CSRF-Token` response header, and re-syncs after reload via `GET /api/auth/csrf`.

The public auth routes (sign-in/up/out, `/auth/csrf`) are **outside** the protected scope, so they are not CSRF-gated. Logout is intentionally not CSRF-protected (low severity; keeps logout always reachable).

## Guards (`auth-hooks.ts`)

| Guard | Effect |
|---|---|
| `requireAuth()` | 401 if no valid session; sets `request.user`. Applied to the whole protected scope. |
| `requireRole('admin')` | 403 unless `user.role === role`. |
| `requirePlan('PRO')` | 403 unless `PLAN_HIERARCHY[plan] ≥` target; admin is privileged. |

`PLAN_HIERARCHY = { FREE: 0, PRO: 1, PREMIUM: 2 }`.

## Local Domain Rules

1. A session token is an opaque session id — treat it as a secret; it grants full account access until expiry. It lives in the `session_token` cookie (and, for non-browser clients, may be sent as a Bearer header — same value). Post-cutover it is **not** returned in the sign-in/sign-up body.
2. Sign-up cannot self-assign a plan (`input: false`).
3. `GET /auth/session` returns `401` when there's no valid session, and resolves authenticated sessions through the same `getSession()` helper as `requireAuth()` (extract `session_token` cookie → inject as Bearer). It does **not** forward the raw request to Better Auth: cross-site, Better Auth never sees our `session_token` cookie, so the SPA's post-refresh revalidation would always fail (the cause of the "logged out on every refresh" bug).
4. The session cache is keyed by raw token; treat the token as a secret and keep it out of logs.
5. **Cookie wins over Bearer.** If both are present, `session_token` is used — so any cookie-authed write is CSRF-gated.
6. CSRF applies **only** to cookie-authed writes. Bearer/non-browser clients (MCP, mobile) are not CSRF-gated.

## Extension Points

- Apply `requirePlan('PRO')` to gate the paid Scout scope at the API layer.
- Add OAuth/social providers via Better Auth plugins.
- Add email verification + password-reset UX (the catch-all already proxies the endpoints).
- Use a short-lived, MCP-scoped token for the chat tool callback instead of the full session token.
- ~~Drop the `token` from the login body~~ ✅ done (post-cutover, 2026-06-14) — the token is set as a cookie and removed from the body. **Note:** the `bearer` plugin must stay — `getSession()`/MCP inject the token as a Bearer header to validate it, so it's load-bearing for the cookie flow, not just legacy compat. The remaining (optional) hardening is to drop the Bearer **fallback** in `extractSessionToken` if no non-browser REST client needs it.
- Evaluate short-lived access tokens + refresh rotation (Better Auth `session.expiresIn` / refresh). Not implemented; non-blocking.
