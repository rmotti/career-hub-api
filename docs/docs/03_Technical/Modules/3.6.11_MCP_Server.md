# 3.6.11 MCP Server

## Responsibility

A self-hosted **Model Context Protocol** server mounted at `POST /mcp` that exposes the user's career data as read-only tools and resources for the OpenAI Responses API. It is the data bridge that lets the [Chat](3.6.10_Chat.md) assistant ground its answers. It authenticates itself (outside the main `requireAuth()` scope) and is never called by the SPA directly.

## Files

```
├── context.ts           McpContext = { userId, sessionToken }
├── rate-limit.ts        per-user MCP limiter

├── resources/

│   ├── playbook.ts      playbook://{saveId}

├── schemas.ts / utils.ts / formatters.ts
└── __tests__/           auth + integration tests
```

## Request Lifecycle (`plugin.ts`)

Per `POST /mcp`:
1. `resolveMcpContext(req)` — bearer token → `mcp:session:<token>` (Redis, 5 min) or Better Auth; else 401. Yields `{ userId, sessionToken }`.
2. `checkRateLimit(userId)` — on limit, 429 + `Retry-After`.
3. Log the tool name (`tools/call` → `params.name`) and duration on response finish.
4. Instantiate a **fresh** `McpServer` (`createMcpServer(ctx)`) and a `StreamableHTTPServerTransport`; pipe the request through; close both on connection close.

A new server/transport per request keeps the handler stateless (see [3.8 Scaling](../3.8_Scaling_and_Performance.md) for the cost note).

## Tools (read-only)

| Tool | Purpose |
|---|---|
| `list_saves` | All the user's saves with last-played date. |
| `get_active_save_context` | Active save: club, season, budget, balance. |
| `get_finances` | Budget, balance, total wage bill, squad size. |
| `get_season_performance` | Team stats + top 5 scorers/assisters for a season. |
| `analyze_squad_by_position` | Active roster grouped by sector (GK/DEF/MID/ATT). |
| `identify_squad_gaps` | Formation-relative gaps by severity (delegates to `scouting.service`). |
| `search_transfer_targets` | Up to 20 dataset players by criteria (delegates to `scouting.service`). |
| `evaluate_signing_fit` | Cost/OVR verdict for a `sofifaId` (delegates to `scouting.service`). |

Each tool takes an optional `saveId` (resolved via `resolveSaveId(ctx.userId, saveId)` — defaults to the most recent save), queries **strictly under `ctx.userId`**, and returns Markdown text. Tools that touch the FC26 dataset reuse the dataset/fit-score caches.

## Resources

| Resource | Purpose |
|---|---|
| `playbook://{saveId}` | The save's scout playbook (objective, ideal age band, value/wage caps, weights) — lets the model reason about what counts as a "gap." |
| `save://{saveId}/dossier` | A consolidated dossier of the save. |

## Local Domain Rules

1. **Read-only.** No tool mutates state — which is why `require_approval: 'never'` (set by the chat) is acceptable today. See [3.7 Security #7](../3.7_Security.md#7-mcp-require_approval-never--low-currently-safe).
2. Self-authenticating: `/mcp` is registered outside the protected scope and resolves its own bearer token (cached `mcp:session:<token>`, 5 min).
3. Every tool/resource is scoped to `ctx.userId` — no cross-tenant access.
4. Tool `description`s are the contract the model reads to decide when to call — keep them precise.
5. The only intended caller is OpenAI's tool-use loop, reached via the [Chat](3.6.10_Chat.md) module.

## Extension Points

- More read tools (transfer history, trophy cabinet, OVR trajectory) — register in `tools/index.ts`.
- **Any write tool** must add an authorization re-check and require approval (do not inherit `require_approval: 'never'`).
- Reuse the MCP tools to back non-chat analytics endpoints.

## Auto-audit

- Lifecycle (auth → rate limit → per-request server/transport) read from `mcp/plugin.ts`.
- Self-auth + 5-min cache read from `mcp/auth.ts`.
- Tool/resource inventory read from `mcp/tools/*` and `mcp/resources/*`; user-scoping confirmed in each handler.
