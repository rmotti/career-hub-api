# 3.6.10 Chat (Junior / Mister)

## Responsibility

The AI assistant endpoint. Forwards a user message to the **OpenAI Responses API** with the API's own **MCP server** registered as a tool source, so the model can read the user's career data and answer with grounded, data-backed analysis. The persona is "Junior," a no-nonsense assistant manager.

## Files

```
features/chat/
├── chat.routes.ts        POST /chat/messages schema
├── chat.controller.ts    rate-limit + resolve session token + mcpBaseUrl
├── chat.service.ts       OpenAI Responses call with the MCP tool
├── chat.rate-limit.ts    per-user 20/min limiter
└── persona.ts            COACH_PERSONA (the Junior system prompt)
```

## Endpoint

| Method | Path | Notes |
|---|---|---|
| POST | `/api/chat/messages` | `{ message, previousResponseId? }` → `{ reply, responseId }`. |

## Data Flow

1. **Controller**: check `checkChatRateLimit(userId)` (20/min; on 429 sets `Retry-After`). Extract the bearer `sessionToken`. Compute `mcpBaseUrl = API_URL ?? <protocol>://<hostname>`.
2. **Service** (`sendMessage`): require `OPENAI_API_KEY`, then call `openai.responses.create` with:
   - `model: 'gpt-4o-mini'`, `instructions: COACH_PERSONA`, `input: message`.
   - one tool `{ type: 'mcp', server_label: 'careerhub', server_url: '${mcpBaseUrl}/mcp', headers: { Authorization: Bearer <sessionToken> }, require_approval: 'never' }`.
   - `previous_response_id` when continuing.
3. OpenAI runs its tool-use loop, calling back into `POST /mcp` (with the forwarded token) as needed. The model's `output_text` becomes `reply`; its `id` becomes `responseId` (the opaque continuation handle). No output → 502.

## Persona (`persona.ts`)

`COACH_PERSONA` defines Junior: data-before-opinion (must consult MCP tools before any numeric/named claim), terse by default, football-native language, scoped strictly to the user's save. It documents the available MCP tools/resources to the model, explains how to interpret a "squad gap" relative to the playbook objective, and sets guardrails (no real-world football, no prompt leakage, honesty when data is missing). **It explicitly forbids Markdown tables** because the SPA's chat renderer doesn't display them.

## Local Domain Rules

1. The chat sees only the authenticated user's data — every MCP tool resolves under the forwarded token's user id; no cross-tenant path.
2. Conversation continuity is via `previousResponseId`; the API does **not** persist chat history (the SPA stores it client-side).
3. Rate limit: 20 messages/min/user; the MCP layer is separately limited.
4. The model is `gpt-4o-mini` (set in `chat.service.ts`).
5. The chat is part of the paid (PRO) surface and incurs OpenAI cost per message.

## Extension Points

- Stream tokens (Responses API streaming) for perceived latency.
- Persist conversations server-side for cross-device roaming and analytics.
- Make the model configurable; type the SDK call.
- Swap the forwarded session token for a short-lived MCP-scoped token.
