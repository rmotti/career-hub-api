# 2.1 Business Rules

Domain rules derived from the Prisma schema and the feature services. Each rule references the code that enforces it. "Enforced" = the API rejects violations; "derived" = the API computes it; "convention" = relied upon but not hard-enforced.

## Identity, Plans, Sessions

### 1. One user, one identity
`User.id` is the Better Auth user id; `User.email` is unique. Sign-up creates the user, an `Account` (password), and an initial `Session`.
_Source_: `prisma/schema.prisma` (`User`, `Account`, `Session`), `features/auth/auth.routes.ts`.

### 2. Plans are FREE / PRO / PREMIUM
Default `FREE` at sign-up (`additionalFields.plan`, `input: false` — clients cannot self-assign a plan on sign-up). `PLAN_HIERARCHY = FREE(0) < PRO(1) < PREMIUM(2)`.
_Source_: `UserPlan` enum, `shared/lib/auth.ts`, `shared/utils/auth-hooks.ts`.

### 3. Plans distinguish the free hub from the paid Scout surface
The free hub (saves, squad, transfers, stats, history) is available to all plans; the Scout surface is the paid tier. Plan-based route gating is provided by `requirePlan()` + `PLAN_HIERARCHY` in `shared/utils/auth-hooks.ts`.
_Source_: `shared/utils/auth-hooks.ts`, `UserPlan` enum.

### 4. Sessions are cached for 5 minutes
`requireAuth()` resolves the bearer token via Better Auth and caches the session in Redis under `session:<token>` for 5 min. Sign-out invalidates that key. A ban or server-side revocation can therefore lag by up to the TTL.
_Source_: `shared/utils/auth-hooks.ts`, `features/auth/auth.routes.ts` (`invalidateSessionCache`).

### 5. Admin role is privileged over plan checks
In `requirePlan()`, a user whose `role === 'admin'` satisfies any plan requirement. Roles come from the Better Auth `admin` plugin.
_Source_: `shared/utils/auth-hooks.ts`, `shared/lib/auth.ts`.

## Saves

### 6. A save is owned by one user
`Save.userId` is required. `getSaveById` returns **404 (not 403)** when the requester is not the owner, to avoid leaking existence.
_Source_: `features/saves/saves.service.ts` (`fetchSaveById`).

### 7. Save-scoped access is checked against the owner
The intended contract for every save-scoped operation is to confirm the requester owns the save (`Save.userId === request.user.id`) before reading or mutating — the `assertSaveAccess` pattern (`scout-playbooks`/`shortlist`/`saved-searches`) and `scouting`'s `getActiveStint`. A non-owned save resolves to 404 (rule 6), not 403.
_Source_: `assertSaveAccess` / `getActiveStint`; `fetchSaveById`.

### 8. A new save starts in 2025/26 with exactly one current club stint
`createSave` sets `currentYear = 2025`, `currentSeason = '2025/26'`, `balance = budget`, and creates one `ClubStint` (`isCurrent = true`) plus one `TeamSeasonStats` per competition of the club's country (+ the optional European competition). (Note: the Prisma column defaults are `2026`/`2026/27`, but `createSave` overrides them.)
_Source_: `features/saves/saves.service.ts` (`createSave`).

### 9. A save always has exactly one current club stint
Exactly one `ClubStint.isCurrent = true` per save at any time. Changing club closes the previous stint (`isCurrent = false`, `endYear = currentYear`) and opens a new one atomically.
_Source_: `features/club-stints/club-stints.service.ts` (`createClubStint`).

### 10. The club must exist in the in-memory club list
Creating a save or stint with an unknown club returns 400. Clubs are a static map in code (`CLUBS_BY_LEAGUE`), not a DB table.
_Source_: `features/clubs/clubs.service.ts` (`clubExists`).

### 11. Advancing the season cascades (derived, transactional)
Setting `currentSeason` via `PATCH /saves/:id` runs, in one transaction:
- **Trophies**: upsert a `Trophy` for each ending-season competition the club won (league `leaguePosition === 1`, or cup `cupResult === 'Campeao'`), keyed at `currentYear`.
- **OVR history**: snapshot each active player's `ovr`/`marketValue` into `PlayerOvrHistory` for the ending season.
- **Ageing**: increment `age` by 1 for active players **and players currently out on loan** under 45 (loaned-out players carry `activeClubStintId = null`, so the aging pass matches them by id — the #4.1 C-001 fix).
- **New stat rows**: create `TeamSeasonStats` (per competition) and `PlayerSeasonStats` (per active player) for the new season.
- **Loan returns**: a loaned-out player returns once his loan's `returnSeason` is reached (`null` = legacy single-season loan → returns at the next advance), re-attached to the stint as `status = Role` with fresh season stats. A player still mid-loan (2-season, `returnSeason` not yet reached) **stays out** and gets a fresh informational `LoanSpellStats` row for the new season (rules 24a, 27a).
- If `budget` is also provided, `balance` is reset to `budget`.
_Source_: `features/saves/saves.service.ts` (`updateSave`).

### 12. Deleting a save deletes everything under it
`onDelete: Cascade` on every child relation (`ClubStint`, `Player`, `Transfer`, `ScoutPlaybook`, `ShortlistItem`, `SavedSearch`, and their descendants). No soft delete, no undo.
_Source_: `prisma/schema.prisma`.

### 13. Money units
`Save.budget`, `Save.balance`, `Player.marketValue`, `Transfer.fee` — **millions of €**. `Player.salary`, `Fc26Player.wage` — **thousands of €**.
_Source_: `shared/utils/currency.ts`.

## Squad

### 14. A player belongs to one save
`Player.saveId` required; cascaded on save delete.

### 15. "Active" means `activeClubStintId` points at the current stint
Released players have `activeClubStintId = null`; loaned-out players also detach (`status = Loan`, `activeClubStintId = null`). `?active=true` lists only players attached to the current stint; `?loaned=true` lists players out on loan from the current stint. **Recall** (`POST /players/:id/recall`) re-attaches a loaned-out player to the current stint as `status = Role` mid-season (ending the loan early) **without** advancing his age; afterward the season-advance loan-return pass naturally skips him. Recall is rejected (400) if the player is not out on loan from the current club.
_Source_: `features/players/players.service.ts` (`recallLoanedPlayer`).

### 16. Shirt numbers are unique within the active squad
A duplicate among active players returns **409 with `code = SHIRT_NUMBER_CONFLICT`** and a human message. Numbers must be 1–99.
_Source_: `features/players/players.service.ts` (`checkShirtNumberConflict`).

### 17. Positions are a fixed enum; alternates cannot duplicate the primary
`Position` ∈ {`GOL, ZAG, MEI, ATA, LD, LE, VOL, MC, ME, MD, PE, PD, SA`}. `alternativePosition.positions[]` is validated to contain only valid, non-duplicate codes excluding the primary.
_Source_: `prisma/schema.prisma`, `validateAlternativePosition`.

### 18. Player status is a fixed enum
`Crucial, Important, Role, Sporadic, Promising, Loan`. `Loan` marks a player currently on loan (in or out). Create/update via the public API accepts the non-`Loan` values; `Loan` is set by the transfer/loan flows.
_Source_: `prisma/schema.prisma`, `players.routes.ts` (enum excludes `Loan` on input), transfer service.

### 19. OVR 40–99, potential 40–99, age 15–45
`ovr` and `potential` are validated to 40–99; age 15–45 (route schema). Season advance only ages players under 45.
_Source_: `players.routes.ts`, `players.service.ts`.

### 20. Bulk import matches by club name and de-dupes by player name
`POST /players/import-fc26` imports every `Fc26Player` whose `club` equals the current stint's club, skipping names already present in the save; default status `Important`; invalid dataset positions are skipped. Returns `{ imported, skipped, total }`.
_Source_: `features/players/players.service.ts` (`importFc26Squad`).

## Transfers

### 21. Transfer types are fixed
`compra, venda, emprestimo_entrada, emprestimo_saida`. Season must match `^\d{4}\/\d{2}$`.
_Source_: `prisma/schema.prisma` (`TransferType`), `transfers.service.ts` (`SEASON_PATTERN`).

### 22. Only purchases and sales move the balance
`compra` subtracts `fee` from `balance`; `venda` adds it. Loans (`emprestimo_*`) never change the balance. The mutation is transactional and the updated `save` is returned.
_Source_: `transfers.service.ts` (`createTransfer`, `updateTransfer`).

### 23. Incoming transfers resolve, reactivate, or create a player
For `compra`/`emprestimo_entrada`: if `playerId` matches, that player is re-attached; else an inactive same-name player is reactivated; else a new `Player` is created with defaults (`position MEI`, `age 25`, `ovr 70`). Status becomes `Role` (purchase) or `Loan` (loan-in). Season stats are created.
_Source_: `transfers.service.ts` (`COMPRA_TYPES` branch).

### 24. Outgoing transfers detach the player
For `venda`/`emprestimo_saida` the player's `activeClubStintId` is set to null; loan-out also sets `status = Loan`. Selling/loaning a player not in the active squad is rejected when a `playerId` is supplied. A loan-out additionally records the loan duration (rule 24a) and opens an informational `LoanSpellStats` row for the season (rule 27a).
_Source_: `transfers.service.ts` (`VENDA_TYPES` branch).

### 24a. Loans last 1 or 2 seasons (`returnSeason`)
A `emprestimo_saida` accepts an optional `loanSeasons` of `1` or `2` (default `1`; any other value → 400). The transfer stores a derived `returnSeason` (= loan `season` + `loanSeasons`, in `YYYY/YY`); `null` denotes a legacy single-season loan. On a season advance the loanee returns once `returnSeason` is reached or is null (rule 11). Existing loans are treated as single-season — no backfill needed.
_Source_: `transfers.service.ts` (`createTransfer`, `addSeasons`), `saves.service.ts` (loan-return partition), `prisma/schema.prisma` (`Transfer.returnSeason`).

### 25. Transfers are scoped to the club stint
A transfer records its `clubStintId`. `GET /transfers?season=current` returns only the **current stint's** transfers in the current season, so transfers from a previous club in the same season are excluded. No filter returns the full save history.
_Source_: `transfers.service.ts` (`listTransfers`), `docs/changelog-transfers-club-stint-scope.md`.

## Stats and Trophies

### 26. Stat rows are auto-created
On save creation: one `TeamSeasonStats` per competition of the club's country. On season advance: new `TeamSeasonStats` per competition + `PlayerSeasonStats` per active player. Adding a player creates its current-season stats row.
_Source_: `saves.service.ts`, `players.service.ts`.

### 27. `goalContributions` is derived, never stored
Every stats projection adds `goalContributions = goals + assists`. The `PATCH .../stats` endpoint rejects it as an input field.
_Source_: `players.service.ts`, `players.routes.ts` (`additionalProperties: false`).

### 27a. Loan-spell stats are stored apart and never aggregate
Goals/assists/matches a player records while **out on loan** live in their own `LoanSpellStats` table (unique per `(player, season)`), decoupled from `PlayerSeasonStats`. They are **informational only**: no aggregation reads them — not the player's career history/totals, not the History-tab records/rankings, not the club's `TeamSeasonStats`. A row is opened on loan-out and again for each new season a 2-season loanee stays out (rules 24, 11). Read via `GET /players/:id/loan-stats`; upserted via `PATCH /players/:id/loan-stats`, allowed **only while the player is out on loan** (else 400). They live in a separate table — not a flag on `PlayerSeasonStats` — precisely so an aggregation can never leak them.
_Source_: `players.service.ts` (`getLoanSpellStats`/`upsertLoanSpellStats`), `transfers.service.ts`, `saves.service.ts`, `prisma/schema.prisma` (`LoanSpellStats`).

### 28. Team stats uniqueness and result fields
`@@unique([clubStintId, competitionId, season])`. League competitions use `leaguePosition` (≥1); cups/European use `cupResult`.
_Source_: `prisma/schema.prisma` (`TeamSeasonStats`), `team-stats.service.ts`.

### 29. Trophies are unique per (stint, competition, year)
`@@unique([clubStintId, competitionId, year])`. Manual trophy creation attaches to the **current** stint and requires a valid `competitionId`.
_Source_: `prisma/schema.prisma`, `trophies.service.ts`.

### 30. Cup results are a fixed enum
`Campeao, Final, Semifinal, Quartas, OitavasOuFaseDeGrupos, Eliminado, NaoParticipou`.
_Source_: `prisma/schema.prisma` (`CupResult`).

### 31. Competitions are fixed reference data
Seeded competitions (League / NationalCup / SuperCup / EuropeanCup, with optional `country`); name is unique. Cached 24h in Redis. The "European" subset (Champions/Europa/Conference) drives the season-rollover dropdown.
_Source_: `competitions.service.ts`, `prisma/schema.prisma` (`Competition`, `CompetitionType`).

## Scout

### 32. The FC26 dataset is read-only
`Fc26Player` rows are seeded; the API never mutates them. Users attach them via `ShortlistItem`.
_Source_: schema; no write path to `Fc26Player` outside seed scripts.

### 33. Fit score is computed by an external service, fail-open
When a `saveId` is supplied to the dataset listing, the API groups candidates by position group and calls the external fit-score service (per club profile + objective), caching results 6h. If the service is unset, slow (>3s), or errors, scores come back `null` and the listing still succeeds. In the **scout score** (rule 34) an absent/`null` fit now contributes **0** to the blend rather than dropping the component.
_Source_: `fc26-players.service.ts` (`enrichWithFitScore`), `shared/lib/fit-score-client.ts`.

### 34. Scout score is a weighted blend with a default playbook
Components: `overall, age, historicalFit, potential, marketValue, wage`. The default ("Equilibrado") weights **overall 30 / potential 15 / age 15 / historicalFit 15 / marketValue 25** (no `wage`). Each component is scored 0–100 and the result is the **weight-normalized average of the available components** (sorted desc, `null` last); `scoutScoreConfidence` derives from historical-fit availability.

Per-component scoring (B-003, 2026-06-21):
- **overall** — fixed linear map `OVR 50→0, 95→100`.
- **potential** — *upside* `(potential − ovr) / 20`, clamped (not the raw level).
- **age** — younger is better: 100 at age 16, accelerating per-year penalty (k = 2 ≤20, 3 for 21–25, 5 for 26–33, 7 for 34+). No per-objective age band; `objective` now feeds **only** the fit score.
- **historicalFit** — `fitScore × 100`, or **0** when no profile exists (kept in the blend — see rule 33).
- **marketValue** — budget headroom `100 × (1 − value/B)`, `B = maxMarketValue (cap) ?? save.budget`; a player **over the explicit cap is excluded from the list**; no cap and no budget → unavailable (drops).
- **wage** — same headroom against `maxWage`, but **opt-in**: no `maxWage` → unavailable (drops). There is no wage budget in the domain.
_Source_: `scout-playbooks/scout-score.ts`, `scout-playbooks.types.ts` (`DEFAULT_SCOUT_PLAYBOOK`), `scout-playbooks.service.ts`. See #4.4 B-003.

### 35. Exactly one default playbook per save
`@@unique([saveId, name])`. Setting a playbook `isDefault = true` flips all others to false in a transaction. A playbook needs at least one positive weight.
_Source_: `scout-playbooks.service.ts`.

### 36. Shortlist and saved searches are unique per save
`ShortlistItem @@unique([saveId, fc26PlayerId])` (a player shortlisted at most once per save; notes ≤ 500 chars; priority `LOW|MEDIUM|HIGH`). `SavedSearch @@unique([saveId, name])` (name ≤ 80 chars).
_Source_: `prisma/schema.prisma`, `shortlist.service.ts`, `saved-searches.service.ts`.

### 37. Scouting analysis is read-only and formation-relative
`identify_squad_gaps` evaluates the active squad against a formation's ideal/min per position (`4-3-3` default, or `4-2-3-1`), grading gaps `critical|moderate|low`. `evaluateSigningFit` compares a dataset player's cost vs. budget and OVR vs. the best same-position squad player.
_Source_: `scouting/scouting.service.ts`, `scouting/formations.ts`.

## Chat (Junior / Mister)

### 38. The chat sees only the authenticated user's own data
`POST /chat/messages` calls OpenAI Responses with the API's MCP server registered as a tool source, passing the **user's** bearer token as the MCP `Authorization` header. Every MCP tool resolves data under that user id. There is no cross-tenant path.
_Source_: `features/chat/chat.service.ts`, `mcp/auth.ts`, `mcp/tools/*`.

### 39. The chat is conversational via `previousResponseId`
Continuation is opaque: the client passes back the prior `responseId`. The API does not persist chat history (the SPA stores it client-side).
_Source_: `chat.service.ts`, `chat.routes.ts`.

### 40. MCP tools are read-only
The MCP server exposes only read tools (saves, finances, performance, squad, scouting) and read resources (`playbook://`, `save://…/dossier`); `require_approval: 'never'` is safe because no tool mutates state.
_Source_: `mcp/tools/*`, `mcp/resources/*`, `chat.service.ts`.

## Governance

### 41. All non-auth, non-MCP calls require a session
`requireAuth()` guards the protected scope; without a valid bearer token the API returns 401. `/api/auth/*` and `POST /mcp` resolve auth themselves.
_Source_: `app.ts`, `auth-hooks.ts`, `mcp/auth.ts`.

### 42. Rate limits apply at three layers
Better Auth global limiter (window 60s, max 100; disabled by `DISABLE_RATE_LIMIT=true`), a chat limiter (20/min/user), and an MCP limiter. 429 responses carry `Retry-After`.
_Source_: `shared/lib/auth.ts`, `features/chat/chat.rate-limit.ts`, `mcp/rate-limit.ts`.
