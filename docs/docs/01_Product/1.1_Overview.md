# 1.1 Overview

## What It Is

**fc26-career-hub-api** is the backend service for FC Career Hub — a companion app for EA Sports FC 26 Career Mode. It is a stateless HTTP API (Fastify) over a PostgreSQL database and a Redis cache, deployed independently to Railway.

The API owns persistence and business rules for the whole product: accounts and sessions, careers ("saves"), the clubs a manager passes through, the squad, transfers, season statistics, trophies, a read-only FC26 player dataset for scouting, a configurable scout score, and an MCP server that lets an AI assistant ("Junior") read the user's career data.

It is single-tenant per user by design: every domain row ultimately belongs to one `Save`, and every `Save` belongs to one `User`. There is no shared state and no multi-tenant business operation.

## Who Consumes It

- **The first-party React SPA** — the only intended client. It talks exclusively over `/api/*` with a bearer token.
- **OpenAI's Responses API** — calls back into `POST /mcp` (with the user's session token) when the chat assistant needs career data.
- **An external fit-score microservice** — called server-to-server by the API to score FC26 players against a club profile.
- **Swagger UI at `/docs`** — for humans exploring the contract.

## Core Capabilities (current code)

- **Auth & sessions** — email/password sign-up and sign-in via Better Auth, opaque bearer tokens, sessions cached in Redis.
- **Saves** — create/list/update/delete a career; advancing the season runs a server-side cascade (new stat rows, trophy placeholders, ageing, loan returns).
- **Club stints** — the trail of clubs within a save; changing club closes the old stint and opens a new one atomically.
- **Squad** — add/edit/release players, per-season stats, OVR history, bulk import of a club's real FC26 roster.
- **Transfers** — incoming/outgoing/loans that mutate the save's balance transactionally and auto-link or create the player.
- **Team stats & trophies** — per-competition, per-season records; trophies derived on season advance or added manually.
- **Reference data** — an in-memory club list (by league/country) and 19 fixed competitions cached in Redis.
- **FC26 dataset** — a read-only player database with rich attribute filters and optional fit-score enrichment.
- **Scout score** — configurable weighting "playbooks" that rank dataset players for a save.
- **Shortlist & saved searches** — per-save persistence of scouting intent.
- **Scouting analysis** — squad-gap detection by formation, transfer-target search, and per-signing evaluation.
- **Chat assistant** — OpenAI Responses API + the API's own MCP server, scoped to the user's data.

## Core Value (what the API guarantees)

- **Durable memory** — a career survives a console upgrade or a corrupted game file because the canonical state lives in Postgres.
- **Server-authoritative rules** — financial mutations, season cascades, and uniqueness constraints are enforced in transactions, not trusted from the client.
- **Enriched data** — the FC26 dataset and fit-score model give the client information no spreadsheet provides, computed server-side and cached.

## What It Isn't

- Not a match simulator — it records outcomes the user enters; it does not play games.
- Not multiplayer — no collaborator model, no shared saves.
- Not connected to EA's live game state — all gameplay data is user-entered (squad bulk import is from the static FC26 dataset, not a live save file).
- Not a public/partner API — there is no API-key program, versioning scheme, or third-party contract; the SPA is the only intended consumer.

## Plans

The data model carries three tiers (`UserPlan`: `FREE | PRO | PREMIUM`, default `FREE`). `requirePlan()` and a `PLAN_HIERARCHY` live in `shared/utils/auth-hooks.ts` for gating paid features (the Scout surface is the paid tier). Plan policy is centralized in the auth hooks so it can evolve in one place. See [2.1 Business Rules](../02_Domain/2.1_Business_Rules.md).

## State of the Product

A functional, end-to-end MVP is deployed: Railway (API) + Neon (Postgres) + Railway Redis, the FC26 dataset is seeded, the external fit-score service is integrated, and the MCP-driven chat works. Ongoing work focuses on hardening — broader test coverage, cache-invalidation completeness, and centralizing the authorization checks — rather than new capability.
