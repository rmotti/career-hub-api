# 1.2 Roadmap

A directional read of the API codebase — what the architecture is set up to grow into next — rather than a committed delivery plan. Update it when priorities shift.

## Horizon 1 — Hardening

Tighten the path from "works" to "robust in production." Mostly correctness and consistency, not features.

- **Centralize authorization.** Lift a shared `assertSaveAccess(saveId, userId)` helper so every save-scoped service confirms ownership the same way.
- **Centralize plan enforcement.** Apply `requirePlan('PRO')` to the paid Scout surface at the API layer.
- **Complete cache invalidation.** Audit every write path so no key serves stale data within its TTL.
- **Response schemas + serialization.** Add Fastify response schemas to hot endpoints to enable `fast-json-stringify` and a strict output contract.
- **Test coverage of critical flows.** Save-creation cascade, season advance, transfer balance math, shirt-number conflict, ownership checks.
- **Tighten CORS.** Replace wildcard trusted origins with an explicit allow-list.

Outcome: the same API surface, with consistent, centralized guarantees.

## Horizon 2 — Depth

Push from "system of record" toward "career intelligence," reusing the data already persisted.

- **Aggregated analytics endpoints.** Balance-over-seasons, OVR trajectory, scoring trends, club-by-club summaries — computed server-side from `PlayerOvrHistory`, `PlayerSeasonStats`, `TeamSeasonStats`, `Transfer`.
- ~~**Server-side chat history.**~~ ✅ **Shipped** (`59df112`, 2026-06-19): `ChatConversation`/`ChatMessage` persist conversations server-side (was `localStorage`), enabling roaming across devices and usage analytics. See [4.2 F-001](../04_Next_Steps/4.2_Features.md).
- **Streaming chat responses.** Switch `chat.service.ts` to the streaming Responses API and stream tokens to the client (SSE).
- **Richer scouting.** Real `alternatives` in `evaluateSigningFit` (currently `[]`); multi-playbook comparison in one call; more formations than `4-3-3`/`4-2-3-1`.
- **Save export / snapshot.** A JSON dump of a full save (and an import), giving users a backup before the irreversible season advance.
- **Bulk operations.** Batch transfer/stat entry endpoints to remove the one-row-at-a-time friction.

Outcome: the API answers "what happened across this career," not just "what is the current state."

## Horizon 3 — Reach

Make the service usable beyond a single first-party SPA.

- **API versioning.** Introduce `/api/v1` and a deprecation policy before any third party depends on the contract.
- **Webhooks / events.** Emit domain events (season advanced, trophy won) for future integrations or notifications.
- **Public read-only "career page".** A signed, cacheable, unauthenticated projection of a save for sharing.
- **Idempotency keys** on mutating endpoints to make client retries safe.
- **Observability.** Structured metrics (request latency, cache hit-rate, Prisma pool saturation, fit-score timeouts) beyond the current Pino logs.

Outcome: the API can be a platform, not just the SPA's private backend.

## Investment Criteria

For each candidate, evaluate against:

1. **Does it reduce the chance of a correctness or security incident?** (favors Horizon 1)
2. **Does it deepen value from data we already store?** (favors Horizon 2)
3. **Does it open the API to new consumers safely?** (favors Horizon 3)
4. **Is the lift proportionate, and does it reuse existing services rather than fork them?**

Horizon 1 items take priority because the product is live: consistency and hardening compound before new surface area.

## Non-Goals (won't do)

- Multiplayer / collaborative saves (would require a collaborator model on every table).
- Live EA game-state integration (console save-file reading, Companion App, EA web API).
- Match simulation.
- A public partner API with SLAs before versioning + auth-key infrastructure exists.
