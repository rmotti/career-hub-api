# 4.3 Improvements

Backlog of **improvements** — non-functional changes that make already-correct behavior better: performance, reliability, observability, developer experience, refactors, test coverage, and consistency. An improvement is invisible to a *correct* client's contract: same inputs, same outputs, better internals. If a client could observe a behavior change, it belongs in [4.1 Corrections](4.1_Corrections.md), [4.2 Features](4.2_Features.md), or [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md) instead.

This is a **living list**, populated from analysis. Actionable engineering debt is tracked in [3.5 Technical Debt](../03_Technical/3.5_Technical_Debt.md); this file is the broader, forward-looking polish backlog and should cross-link debt items rather than duplicate them.

## Scope

**Belongs here**
- Performance: query tuning, N+1 removal, cache coverage, payload size, serialization (response schemas → `fast-json-stringify`).
- Reliability / ops: retries, timeouts, graceful degradation, idempotency hardening, metrics & alerting.
- DX & maintainability: refactors, dedup, shared helpers, stronger types, file organization.
- Test coverage of already-correct flows; docs accuracy.

**Does not belong here**
- A behavior that is actually *wrong* → [4.1 Corrections](4.1_Corrections.md).
- A new capability a client can call → [4.2 Features](4.2_Features.md).
- A deliberate change in what the rule *means* → [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md).

> **Invariant guard:** any improvement touching state must respect the **stateless-process invariant** (see [3.1 Architecture](../03_Technical/3.1_Architecture.md) / CLAUDE.md). No new mutable in-process state — route caches/counters through Redis. Flag items that risk this.

## Type & status legend

- **Type**: `perf` · `reliability` · `observability` · `dx` · `refactor` · `tests` · `docs`.
- **Effort**: `S` · `M` · `L`.

Status: `proposed` → `accepted` → `in-progress` → `done`.
