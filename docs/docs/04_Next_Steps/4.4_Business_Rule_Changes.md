# 4.4 Business Rule Changes

Backlog of **business rule changes** ‚Äî deliberate changes to the domain's rules, where the old behavior was *correct under the old rule* and we are choosing to redefine it. This is distinct from a defect (the rule was right, the code was wrong ‚Üí [4.1 Corrections](4.1_Corrections.md)) and from a feature (net-new capability that doesn't redefine an existing rule ‚Üí [4.2 Features](4.2_Features.md)). Rule changes are frequently **breaking** and almost always need a migration and/or client coordination.

This is a **living list**, populated from analysis. The current rules are catalogued in [2.1 Business Rules](../02_Domain/2.1_Business_Rules.md); **every item here must name the rule it changes** so 2.1 stays the source of truth once shipped.

## Scope

**Belongs here**
- Redefining a domain rule (e.g., transfer balance math, season-advance side effects, plan gating boundaries, ownership/visibility semantics).
- Changing a contract on purpose: response shape, error code, or default that clients depend on.
- Changing units, defaults, or lifecycle (e.g., what a season advance does, what "current" means).

**Does not belong here**
- The rule is fine but the code violates it ‚Üí [4.1 Corrections](4.1_Corrections.md).
- A new rule for a brand-new capability ‚Üí define it with the [4.2 Feature](4.2_Features.md) and link back.
- Internal-only change with no rule/contract impact ‚Üí [4.3 Improvements](4.3_Improvements.md).

## Impact & status legend

- **Breaking?**: `yes` (clients must change) ¬∑ `no` (additive/compatible) ¬∑ `flagged` (behind a flag/version).
- **Migration**: `none` ¬∑ `data` (backfill) ¬∑ `schema` ¬∑ `data+schema`.

Status: `proposed` ‚Üí `decided` ‚Üí `in-progress` ‚Üí `shipped` ‚Üí `2.1-updated`.

## Business Rule Changes

### B-001 ‚Äî Editable loan-spell stats, excluded from all history
- **Status**: 2.1-updated (shipped + deployed 2026-06-19; migration applied to prod; 2.1 reconciled) ‚úÖ
- **Breaking?**: no (additive)
- **Migration**: schema ‚Äî `LoanSpellStats` table, **decoupled** from `PlayerSeasonStats` (migration `20260619000000_add_loan_spell_stats`)
- **Rule today (from 2.1)**: stats are recorded per `(player, clubStint, season)` via `PlayerSeasonStats`, which is exactly what powers the player's career history (#4.2 F-004), the History-tab records/rankings (#4.2 F-003), and your club totals/`TeamSeasonStats`. A loaned-out player has `activeClubStintId = null`, so today you can't record his goals/assists at all. ([2.1](../02_Domain/2.1_Business_Rules.md))
- **New rule**: while a player is out on loan, allow **editing/recording** his goals/assists at the loan club, but those numbers **must not sum into anything** ‚Äî not his career history (#4.2 F-004), not the History-tab records/rankings (#4.2 F-003), and not your club's totals/`TeamSeasonStats`. They are **informational only** (the loanee's form while away).
- **Why**: follow a loanee's development while he's away, without polluting his career history or your club's records.
- **Affected surface**: a new loan-spell stats store + edit endpoints; squad/loan UI to show them in the loan context only; explicit exclusion from every history/ranking/club aggregation.
- **Client impact & rollout**: additive; existing history/records are unaffected by definition.
- **Backfill / migration plan**: none for existing data; add the new store.
- **Design decision (requirement settled 2026-06-19; mechanism recommended)**: because the numbers must **never** aggregate, store them **fully outside `PlayerSeasonStats`** ‚Äî a dedicated `LoanSpellStats` table keyed by player + loan spell/season. A flag on `PlayerSeasonStats` (e.g. `excludeFromHistory`) is **rejected**: it's leak-prone ‚Äî every aggregation query would have to remember to filter it, and one missed filter silently corrupts records/rankings.
- **Docs to update on ship**: [[2.1 Business Rules]] loan + stats rules ¬∑ changelog ¬∑ OpenAPI.
- **Related**: #4.1 C-001, #4.2 F-002, #4.2 F-003, #4.2 F-004, #4.4 B-002

_Implemented (2026-06-19): `LoanSpellStats` model in [schema.prisma](../../../prisma/schema.prisma); created on loan-out in [transfers.service.ts](../../../src/features/transfers/transfers.service.ts) and for players staying out another season in the season-advance pass ([saves.service.ts](../../../src/features/saves/saves.service.ts)); read/edit via `GET`/`PATCH /saves/:id/players/:pid/loan-stats` ([players.service.ts](../../../src/features/players/players.service.ts), guarded to players currently out on loan). No aggregation query references the table ‚Äî career history, History rankings and `TeamSeasonStats` are untouched by design. Tests: `players.service.test.ts` ("loan-spell stats (B-001)") + `transfers.service.test.ts`. [[2.1 Business Rules]] reconciled (2026-06-19): rule 27a (+ 24, 11). Migration applied to prod 2026-06-19 (`migrate resolve --applied` on the interrupted run, then `deploy`). **Code deployed 2026-06-19 (commit `bd578fc`) ‚Äî fully live.**_

### B-002 ‚Äî Two-season loan support
- **Status**: 2.1-updated (shipped + deployed 2026-06-19; migration applied to prod; 2.1 reconciled) ‚úÖ
- **Breaking?**: no (extends the loan model)
- **Migration**: schema ‚Äî `Transfer.returnSeason` nullable column (migration `20260619000001_add_transfer_return_season`)
- **Rule today (from 2.1)**: loans are effectively single-season ‚Äî on the next season advance, loaned-out players are auto-returned to the current stint (`saves.service.ts` loan-return pass ~L287). ([2.1](../02_Domain/2.1_Business_Rules.md))
- **New rule**: support loans spanning two seasons ‚Äî the player stays out for the configured duration and only returns at the end.
- **Why**: match real Career Mode loan options.
- **Affected surface**: loan creation (`transfers`), season-advance loan-return logic, age accrual (per #4.1 C-001 ‚Äî the player ages each of the two loan seasons, same as squad), recall (#4.2 F-002).
- **Client impact & rollout**: additive; existing single-season loans unaffected.
- **Backfill / migration plan**: add a loan end-season field; default existing loans to one season.
- **Docs to update on ship**: [[2.1 Business Rules]] loan duration/return rules ¬∑ changelog ¬∑ OpenAPI.
- **Related**: #4.1 C-001, #4.2 F-002, #4.4 B-001

_Implemented (2026-06-19): `Transfer.returnSeason` ([schema.prisma](../../../prisma/schema.prisma)); `createTransfer` accepts `loanSeasons` (1 or 2, default 1) and derives `returnSeason` ([transfers.service.ts](../../../src/features/transfers/transfers.service.ts)); the season-advance loan-return pass now partitions loanees into **returning** (`returnSeason` null or reached) vs **still on loan** ([saves.service.ts](../../../src/features/saves/saves.service.ts)). `null` = legacy single-season loan, so existing loans need no backfill. Players staying out age via #4.1 C-001 and get a fresh #4.4 B-001 loan-spell row. Tests: `saves.service.test.ts` (2-season) + `transfers.service.test.ts`. [[2.1 Business Rules]] reconciled (2026-06-19): rules 24a, 11, 15. Migration applied to prod 2026-06-19. **Code deployed 2026-06-19 (commit `bd578fc`) ‚Äî fully live.**_
