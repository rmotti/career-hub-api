# 4.4 Business Rule Changes

Backlog of **business rule changes** — deliberate changes to the domain's rules, where the old behavior was *correct under the old rule* and we are choosing to redefine it. This is distinct from a defect (the rule was right, the code was wrong → [4.1 Corrections](4.1_Corrections.md)) and from a feature (net-new capability that doesn't redefine an existing rule → [4.2 Features](4.2_Features.md)). Rule changes are frequently **breaking** and almost always need a migration and/or client coordination.

This is a **living list**, populated from analysis. The current rules are catalogued in [2.1 Business Rules](../02_Domain/2.1_Business_Rules.md); **every item here must name the rule it changes** so 2.1 stays the source of truth once shipped.

## Scope

**Belongs here**
- Redefining a domain rule (e.g., transfer balance math, season-advance side effects, plan gating boundaries, ownership/visibility semantics).
- Changing a contract on purpose: response shape, error code, or default that clients depend on.
- Changing units, defaults, or lifecycle (e.g., what a season advance does, what "current" means).

**Does not belong here**
- The rule is fine but the code violates it → [4.1 Corrections](4.1_Corrections.md).
- A new rule for a brand-new capability → define it with the [4.2 Feature](4.2_Features.md) and link back.
- Internal-only change with no rule/contract impact → [4.3 Improvements](4.3_Improvements.md).

## Impact & status legend

- **Breaking?**: `yes` (clients must change) · `no` (additive/compatible) · `flagged` (behind a flag/version).
- **Migration**: `none` · `data` (backfill) · `schema` · `data+schema`.

Status: `proposed` → `decided` → `in-progress` → `shipped` → `2.1-updated`.

## Business Rule Changes

### B-001 — Editable loan-spell stats, excluded from all history
- **Status**: 2.1-updated (shipped + deployed 2026-06-19; migration applied to prod; 2.1 reconciled) ✅
- **Breaking?**: no (additive)
- **Migration**: schema — `LoanSpellStats` table, **decoupled** from `PlayerSeasonStats` (migration `20260619000000_add_loan_spell_stats`)
- **Rule today (from 2.1)**: stats are recorded per `(player, clubStint, season)` via `PlayerSeasonStats`, which is exactly what powers the player's career history (#4.2 F-004), the History-tab records/rankings (#4.2 F-003), and your club totals/`TeamSeasonStats`. A loaned-out player has `activeClubStintId = null`, so today you can't record his goals/assists at all. ([2.1](../02_Domain/2.1_Business_Rules.md))
- **New rule**: while a player is out on loan, allow **editing/recording** his goals/assists at the loan club, but those numbers **must not sum into anything** — not his career history (#4.2 F-004), not the History-tab records/rankings (#4.2 F-003), and not your club's totals/`TeamSeasonStats`. They are **informational only** (the loanee's form while away).
- **Why**: follow a loanee's development while he's away, without polluting his career history or your club's records.
- **Affected surface**: a new loan-spell stats store + edit endpoints; squad/loan UI to show them in the loan context only; explicit exclusion from every history/ranking/club aggregation.
- **Client impact & rollout**: additive; existing history/records are unaffected by definition.
- **Backfill / migration plan**: none for existing data; add the new store.
- **Design decision (requirement settled 2026-06-19; mechanism recommended)**: because the numbers must **never** aggregate, store them **fully outside `PlayerSeasonStats`** — a dedicated `LoanSpellStats` table keyed by player + loan spell/season. A flag on `PlayerSeasonStats` (e.g. `excludeFromHistory`) is **rejected**: it's leak-prone — every aggregation query would have to remember to filter it, and one missed filter silently corrupts records/rankings.
- **Docs to update on ship**: [[2.1 Business Rules]] loan + stats rules · changelog · OpenAPI.
- **Related**: #4.1 C-001, #4.2 F-002, #4.2 F-003, #4.2 F-004, #4.4 B-002

_Implemented (2026-06-19): `LoanSpellStats` model in [schema.prisma](../../../prisma/schema.prisma); created on loan-out in [transfers.service.ts](../../../src/features/transfers/transfers.service.ts) and for players staying out another season in the season-advance pass ([saves.service.ts](../../../src/features/saves/saves.service.ts)); read/edit via `GET`/`PATCH /saves/:id/players/:pid/loan-stats` ([players.service.ts](../../../src/features/players/players.service.ts), guarded to players currently out on loan). No aggregation query references the table — career history, History rankings and `TeamSeasonStats` are untouched by design. Tests: `players.service.test.ts` ("loan-spell stats (B-001)") + `transfers.service.test.ts`. [[2.1 Business Rules]] reconciled (2026-06-19): rule 27a (+ 24, 11). Migration applied to prod 2026-06-19 (`migrate resolve --applied` on the interrupted run, then `deploy`). **Code deployed 2026-06-19 (commit `bd578fc`) — fully live.**_

### B-002 — Two-season loan support
- **Status**: 2.1-updated (shipped + deployed 2026-06-19; migration applied to prod; 2.1 reconciled) ✅
- **Breaking?**: no (extends the loan model)
- **Migration**: schema — `Transfer.returnSeason` nullable column (migration `20260619000001_add_transfer_return_season`)
- **Rule today (from 2.1)**: loans are effectively single-season — on the next season advance, loaned-out players are auto-returned to the current stint (`saves.service.ts` loan-return pass ~L287). ([2.1](../02_Domain/2.1_Business_Rules.md))
- **New rule**: support loans spanning two seasons — the player stays out for the configured duration and only returns at the end.
- **Why**: match real Career Mode loan options.
- **Affected surface**: loan creation (`transfers`), season-advance loan-return logic, age accrual (per #4.1 C-001 — the player ages each of the two loan seasons, same as squad), recall (#4.2 F-002).
- **Client impact & rollout**: additive; existing single-season loans unaffected.
- **Backfill / migration plan**: add a loan end-season field; default existing loans to one season.
- **Docs to update on ship**: [[2.1 Business Rules]] loan duration/return rules · changelog · OpenAPI.
- **Related**: #4.1 C-001, #4.2 F-002, #4.4 B-001

_Implemented (2026-06-19): `Transfer.returnSeason` ([schema.prisma](../../../prisma/schema.prisma)); `createTransfer` accepts `loanSeasons` (1 or 2, default 1) and derives `returnSeason` ([transfers.service.ts](../../../src/features/transfers/transfers.service.ts)); the season-advance loan-return pass now partitions loanees into **returning** (`returnSeason` null or reached) vs **still on loan** ([saves.service.ts](../../../src/features/saves/saves.service.ts)). `null` = legacy single-season loan, so existing loans need no backfill. Players staying out age via #4.1 C-001 and get a fresh #4.4 B-001 loan-spell row. Tests: `saves.service.test.ts` (2-season) + `transfers.service.test.ts`. [[2.1 Business Rules]] reconciled (2026-06-19): rules 24a, 11, 15. Migration applied to prod 2026-06-19. **Code deployed 2026-06-19 (commit `bd578fc`) — fully live.**_

### B-003 — Scout scoring engine redesign (cost, age, fit, potential, default weights)
- **Status**: 2.1-updated (shipped to main 2026-06-21; rule 34/33 + 3.6.9 reconciled)
- **Breaking?**: yes — `scoutScore` values and the `/scout/evaluate` ranking change for every client
- **Migration**: none (the score is computed, never stored). The default playbook weights change (rule 34); existing **custom** playbooks keep their explicit weights — only the *meaning* of each component score shifts.
- **Rule today (from 2.1)**: rule 34 — scout score is the weight-normalized average of *available* components; default "Equilibrado" = overall 35 / age 20 / historicalFit 25 / potential 20, with cost (`marketValue`/`wage`) **absent** from the default. Per rule 33, unavailable fit drops the component and renormalizes the remaining weights. ([2.1](../02_Domain/2.1_Business_Rules.md))
- **New rule** — redefine how each metric scores so the engine answers *"best signing for the level and what I'll spend"*:
  1. **Cost is budget-relative, not absolute.** Within budget the score is linear headroom `score = 100 × (1 − value/B)` (free → 100, at-budget → 0); the **hard filter** excludes a player over the *explicit cap* from the list (consistent with `findAlternatives`, `scouting.service.ts:163-168`). The two cost axes are **asymmetric** because only one has a real reference in the domain:
     - **`marketValue`** → `B = maxMarketValue (cap) ?? save.budget`. Works by default (the save *has* a transfer budget). No cap + null budget → component **unavailable, drops** (see #3 rationale).
     - **`wage`** → `B = maxWage (cap)` **only** — there is no wage-budget field in the domain, so wage is **opt-in**: no `maxWage` → component **unavailable, drops**. No invented constant, no candidate-distribution fallback (that would make a wage score depend on who else is in the search — the same relativity we rejected for `overall`).

     Replaces the log + magic-22 — carries #4.1 C-002.
  2. **Age is a fixed convex curve, no objective bands.** Younger is better and the penalty accelerates: per-year `k` = 2 (≤20), 3 (21–25), 5 (26–33), 7 (34+), anchored 100 @ age 16 → e.g. 18→96, 25→77, 30→52, 34→30, 39+→0. Drops `DEFAULT_OBJECTIVE_AGE_RANGE` and the per-user `idealAgeMin/idealAgeMax` (the curve **is** the model; `maxAge` stays as a hard filter). Weight = intent (high weight → wants youth; 0 → age doesn't matter).
  3. **Fit unavailable = score 0** (not renormalized away), so the weight stays in the denominator and rankings stay comparable across players. (Accepted trade-off: penalizes missing fit-data as if it were a bad fit; the user will improve fit-data coverage separately — interacts with rule 33.)
  4. **Potential = the real ceiling, on the same fixed scale as `overall`**: `score = clamp((potential − 50) / 45 × 100)`. _(Corrected 2026-06-21 — **reverted from an earlier "upside" `(potential − ovr)` design**, which perversely rewarded weak players with headroom: an OVR 55 / potential 70 (gap 15) outscored an OVR 88 / potential 90 (gap 2). The youth dimension is already carried by the **age** component, so potential must measure the ceiling, not the gap.)_ Potential distribution verified 2026-06-21: min 49 / max 95 / p50 71 / p99 86 — the `95` ceiling **is** the real dataset max (so a max-potential prospect scores 100).
  5. **Normalize `overall`/`potential` to the real 0–100 band** via a **fixed** linear map `OVR 50 → 0`, `95 → 100` (`score = clamp((ovr − 50) / 45 × 100)`) — same scale for both components. Anchors calibrated to the **real dataset distribution** (18,405 players: min 47, max 91, median 66, p90 75, p99 83): floor 50 because sub-50 is effectively empty; top 95 = just above the real max (91) so the GOAT scores ≈91 with headroom, and the upper band isn't dead (a `99→100` anchor would cap the best player at ≈84 and waste 84–100). Fixed, **not** relative to the candidate list (a relative min/max would stretch a narrow search, e.g. an 85+ band, so an OVR 85 scores 0 — false). Resulting scale: 66 (median) ≈ 36, 80 ≈ 67, 85 ≈ 78, 91 (max) ≈ 91 — comparable across searches (an "85" scores the same every time) and truthful about magnitude (a 5-OVR gap is a small gap, so cost/age/fit break ties between similar-level players). Today the raw OVR sits in a narrow high band, so the *effective* weight ≠ the *configured* weight; the fixed scale fixes that. _(Dataset facts verified 2026-06-21 against prod `Fc26Player`.)_
  6. **New default "Equilibrado" weights**: overall 30 / potential 15 / age 15 / historicalFit 15 / marketValue 25 (cost = 25%, all via `save.budget`). **No `wage` in the default** — it can't score without a `maxWage` (point 1), so it's a pure opt-in component the user adds when salary matters.
- **Why**: the current scoring has metrics that are miscalibrated or point the wrong way (cost mixes units #4.1 C-002; age uses a band that *penalizes* youth; potential rewards level not upside; unavailable fit silently re-weights the blend; raw-value components make weights lie). The redesign makes the score a coherent value-for-money signal.
- **Affected surface**: `scout-score.ts` (`scoreBudget`, `scoreAge`, `calculateComponent`, `calculateScoutScore`, `inferScoreConfidence`), `scout-playbooks.types.ts` (`DEFAULT_SCOUT_PLAYBOOK`; remove `idealAgeMin/idealAgeMax` from `ScoutPlaybookPreferences`), `scout-playbooks.service.ts` (thread `save.budget` into the eval context + apply the over-cap filter), `/scout/evaluate` output, and the MCP `playbook://` resource render.
- **Client impact & rollout**: ranking + `scoutScore`/breakdown values change for all callers; the FE Scout surface shifts. Best shipped together with the #4.2 F-007 shaping. `objective` survives **only** as the fit-score input (it no longer touches age scoring).
- **Backfill / migration plan**: none (computed). `idealAgeMin/idealAgeMax` and the per-objective age range become dead config — safe to ignore on read; drop from the create/update schema.
- **Docs to update on ship**: [[2.1 Business Rules]] rule 34 (and note the rule 33 interaction — null fit now scores 0 in the blend) · [[3.6.9 Scout]] scoring engine section · changelog · OpenAPI.
- **Related**: #4.1 C-002 (cost unit defect, bundled) · #4.2 F-007 (Scout EPIC) · [[2.1 Business Rules]] rules 33, 34

_Implemented (2026-06-21): engine rewrite in [scout-score.ts](../../../src/features/scout-playbooks/scout-score.ts) — `scoreLevel` (rating 50→0/95→100, used for **both** `overall` and `potential`), `scoreAge` (convex `AGE_BANDS` k=2/3/5/7, 100@16), `scoreBudget` (`100×(1−value/B)`), fit-absent → score 0 (kept, not dropped), and a new `ScoutScoreContext` ({ marketValueRef, wageRef }) threaded through `calculateScoutScore`/`calculateComponent`. FE adaptation handed off (breaking score/ranking change).**_
