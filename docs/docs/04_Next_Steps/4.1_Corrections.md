# 4.1 Corrections

Backlog of **corrections** — defects where the API's actual behavior diverges from its *intended* behavior. A correction restores the contract that already exists; it adds no new capability (→ [4.2 Features](4.2_Features.md)), changes no domain rule on purpose (→ [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md)), and is not a non-functional polish of already-correct behavior (→ [4.3 Improvements](4.3_Improvements.md)).

This is a **living list**, populated from analysis. Order by severity, then by blast radius.

## Scope

**Belongs here**
- Wrong results, off-by-one math, incorrect status codes, stale cache served past intent.
- Crashes / unhandled errors / 500s that should be a typed error.
- Validation gaps that let invalid state through, or valid input rejected.
- Authorization/ownership leaks where the *intended* rule is already "deny."
- Contract drift: response shape or error envelope not matching the documented contract.

**Does not belong here**
- New endpoints or net-new behavior → [4.2 Features](4.2_Features.md).
- "Works, but could be faster / cleaner / better tested" → [4.3 Improvements](4.3_Improvements.md).
- A deliberate change to a domain rule (the old behavior was *correct per the old rule*) → [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md).

## Severity legend

- **S1 — Critical**: data corruption, auth/ownership leak, money math wrong, or a down endpoint.
- **S2 — High**: wrong result on a common path, but recoverable / scoped.
- **S3 — Medium**: edge-case defect, wrong status code, misleading message.
- **S4 — Low**: cosmetic / rare, no data impact.

Status: `proposed` → `accepted` → `in-progress` → `fixed` → `verified`.

## Corrections

### C-001 — Player age does not advance when returning from loan
- **Severity**: S2
- **Status**: fixed (done)
- **Area / files**: `features/saves/saves.service.ts` — season-advance aging pass (~L253) and loan-return pass (~L287)
- **Expected**: every player ages +1 per season advance, including a player who spent the season(s) out on loan and is returned during this advance.
- **Actual**: the aging `UPDATE` filters `activeClubStintId = currentStint.id` and runs **before** the loan-return block re-attaches loaned players (which carry `activeClubStintId = null` while out). Loaned-out players are skipped by the aging pass for every loan season and the return season.
- **Repro**: loan a player out → advance one (or more) seasons → player returns with the same age he left with.
- **Root cause**: ordering + filter — a loaned-out player has `activeClubStintId = null`, so no aging pass touches him; on return the aging has already run.
- **Fix sketch**: include loaned-out players in the season-advance aging pass so they age **+1 per advance, identical to squad players** (decided 2026-06-19 — no special catch-up logic). Extend the aging `UPDATE` to also cover `status = Loan` rows, not just `activeClubStintId = currentStint.id`.
- **Regression test**: season-advance test asserting a loaned-out player's age increments by the number of seasons advanced.
- **Related**: [[2.1 Business Rules]] aging rule · interacts with #4.4 B-002 (2-season loans must age correctly) and #4.2 F-002 (recall)

_Fixed (2026-06-19): the season-advance aging pass now ages the active squad **and** loaned-out players in a single `updateMany` (loaned-out players matched by `id`, since they carry `activeClubStintId = null`); the loan-return block reuses the loaned set fetched before aging — see [saves.service.ts](../../../src/features/saves/saves.service.ts). Regression guard added in `saves.service.test.ts` ("returns loaned-out players to the squad on advance, and ages them (C-001)"). Still `proposed`→`fixed`; `verified` pending an integration test against a real DB (disposable local Postgres)._
