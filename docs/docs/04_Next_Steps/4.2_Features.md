# 4.2 Features

Backlog of **features** — net-new capability the API does not have today: new endpoints, new resources, new integrations, or a meaningfully new behavior on an existing surface. A feature *adds* what isn't there; it does not fix broken behavior (→ [4.1 Corrections](4.1_Corrections.md)), polish existing behavior (→ [4.3 Improvements](4.3_Improvements.md)), or redefine a current domain rule (→ [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md)).

This is a **living list**, populated from analysis. The directional themes live in [1.2 Roadmap](../01_Product/1.2_Roadmap.md); this file is the concrete, item-level backlog those themes resolve into.

## Scope

**Belongs here**
- New endpoints / resources (e.g., aggregated analytics, export/snapshot, bulk operations).
- New integrations or external surfaces (e.g., streaming chat, new MCP tools).
- A new capability layered onto an existing module that a correct client could not do before.

**Does not belong here**
- Restoring intended-but-broken behavior → [4.1 Corrections](4.1_Corrections.md).
- Making an existing capability faster / cleaner / better covered → [4.3 Improvements](4.3_Improvements.md).
- Changing the meaning of an existing rule or contract → [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md).

## Sizing & status legend

- **Size**: `S` (≤1 day) · `M` (a few days) · `L` (a week+) · `XL` (multi-week / needs breakdown).
- **Tier impact**: which plan it touches (FREE hub vs PRO Scout surface — see [2.1 Business Rules](../02_Domain/2.1_Business_Rules.md#identity-plans-sessions)).

Status: `idea` → `shaped` → `accepted` → `in-progress` → `shipped`.

## Features

### F-001 — Server-side chat history persistence
- **Status**: shipped (2026-06-19)
- **Size**: M
- **Tier**: PRO (chat is PRO)
- **Problem / motivation**: chat history lived in the SPA's `localStorage` — no roaming across devices, no continuity, no usage analytics.
- **Surface (implemented)**:
  - `POST /chat/conversations` — create a conversation (`{ title?, saveId? }`); user-scoped, optionally save-scoped.
  - `GET /chat/conversations` — list conversations (`?saveId=` filter); ordered by `updatedAt desc`.
  - `DELETE /chat/conversations/:conversationId` — delete conversation + cascade messages.
  - `GET /chat/conversations/:conversationId/messages` — list messages ordered `createdAt asc`.
  - `POST /chat/messages` — extended with optional `conversationId`: when set, persists the user turn + assistant reply and auto-derives `previousResponseId` from the last assistant message. Backward-compatible: omitting `conversationId` works exactly as before.
- **Acceptance criteria**:
  - [x] messages survive reload and device switch (persisted in `ChatMessage`, read via GET)
  - [x] history scoped to the user (and PRO-gated via existing scope in `app.ts`)
- **Data / schema impact**: `ChatRole` enum + `ChatConversation` + `ChatMessage` models; migration `20260619000002_add_chat_conversations`. `ChatConversation.saveId` FK → `Save` (`SET NULL` on delete). `ChatMessage` FK → `ChatConversation` (`CASCADE`).
- **Key files**: `features/chat/conversations.service.ts` (CRUD + persistence), `chat.controller.ts`, `chat.routes.ts`; `prisma/schema.prisma`.
- **Answers U5**: **shortlist and saved searches already persist** — `ShortlistItem` and `SavedSearch` Prisma models exist with services. Only chat history was missing, now closed.
- **Related**: [[1.2 Roadmap]] H2 · pairs with #4.3 I-005 (persona)

### F-002 — "Recall" a loaned-out player before the new season
- **Status**: shipped (deployed 2026-06-19, commit `bd578fc`) ✅
- **Size**: M
- **Tier**: FREE (the squad/loan hub is free)
- **Problem / motivation**: a loaned-out player can only come back on the next season advance; the user wants to recall mid-season.
- **Surface (implemented)**: `POST /saves/:saveId/players/:playerId/recall` — re-attaches the player to the current stint (`status=Role`), ends the loan early, and ensures his `PlayerSeasonStats` for the current season. Audited as `player.loan_recall`.
- **Acceptance criteria**:
  - [x] player returns to the squad immediately
  - [x] does not age on recall (aging happens only on a season advance — decided 2026-06-19)
  - [x] no double-return: the season-advance loan-return pass naturally skips him (no longer `status=Loan` + `activeClubStintId=null`)
  - [x] loan duration honored: `Transfer.returnSeason` now exists (#4.4 B-002); recall ends the loan by flipping the player back to `Role` (the season-advance pass then skips him), so no separate "ended-early" flag is needed
  - [x] informational loan-spell stats preserved on recall — recall never touches `LoanSpellStats` rows (#4.4 B-001)
- **Data / schema impact**: none for F-002 itself (no migration). Loan-end duration ships via #4.4 B-002; loan-spell stats via #4.4 B-001.
- **Dependencies**: builds on #4.1 C-001 (aging, done); coordinates with #4.4 B-001 (loan stats, done) and #4.4 B-002 (2-season, done).
- **Related**: #4.4 B-001, #4.1 C-001, #4.4 B-002

_Implemented (2026-06-19): service `recallLoanedPlayer` in [players.service.ts](../../../src/features/players/players.service.ts), controller + `POST .../recall` route in [players.routes.ts](../../../src/features/players/players.routes.ts). Guards: 400 if the player is not on loan from the current club. Tests in `players.service.test.ts` ("recallLoanedPlayer (F-002)"). No DB migration. **Deployed 2026-06-19 (commit `bd578fc`) — fully live.**_

### F-003 — History: show the clubs a ranked player played for with you
- **Status**: shipped (API 2026-06-20 + FE integrated 2026-06-20) ✅
- **Size**: S
- **Tier**: FREE
- **Problem / motivation**: the history ranking shows the player but not the clubs (stints) he featured for across your save.
- **Surface (implemented)**: the default list variant `GET /saves/:saveId/players` (the one that returns `totalStats` and powers the History ranking) now also returns `clubs: string[]` — the **distinct** clubs the player featured for across the save, in **first-appearance order** (chronological by season). The `?active` and `?loaned` variants are unchanged (no `clubs`). The SPA History ranking consumes `clubs` and displays them next to each ranked player.
- **Acceptance criteria**:
  - [x] each ranked player shows the clubs (stints) he appeared for with you (`clubs[]`) — *API + FE shipped*
- **Data / schema impact**: none — derived from `PlayerSeasonStats.clubStint.club`; read/aggregate only, no migration.
- **Design decision (2026-06-20)**: `clubs` is a flat **list of names** (`string[]`), distinct and chronologically ordered — not per-club stat objects (per-club breakdown stays in the player detail view, #4.2 F-004). Same club across multiple stints/seasons collapses to a single entry.
- **Key files**: `features/players/players.service.ts` (`fetchPlayers`, default branch — third parallel query + in-memory dedup), `players.routes.ts` (`playerListItemResponse` schema + route description); test in `players.service.test.ts` ("listPlayers — ranking clubs (F-003)"). FE: History ranking in the companion SPA repo (integrated + committed 2026-06-20).
- **Related**: #4.3 I-004 (history top-N) · #4.2 F-004 (per-club stats live in player detail)

### F-004 — Full per-season player history in squad detail
- **Status**: shipped (API 2026-06-19 + FE integrated) ✅
- **Size**: M
- **Tier**: FREE
- **Problem / motivation**: the squad "eye"/detail view shows only the last-season OVR / market-value delta, not the full trajectory or per-season stats.
- **Surface (implemented)**: the existing `GET /saves/:saveId/players/:playerId` was **enriched** rather than adding a new endpoint — it already returned `history` (per season+club) and `ovrHistory` (OVR/MV per season); now also returns:
  - `seasons[]` — **merged, render-ready** per-season view: `{ season, ovr|null, marketValue|null, clubs: [{ club, goals, assists, matches, cards…, goalContributions }] }`, sorted chronologically. `clubs[]` handles a mid-season club change (two `PlayerSeasonStats` rows for one season); a season with an OVR snapshot but no stats still appears (`clubs: []`). This is the primary source for the FE (no client-side merge needed).
  - `loanSpells[]` — `{ season, loanClub, goals, assists, matches, goalContributions }`, surfaced as a **separate section**, never merged into `seasons`/`history`/`totalStats` (#4.4 B-001, informational only).
  - `history`/`ovrHistory` kept for backward compat; `history` (seasonStats) now `orderBy season asc`.
- **Acceptance criteria**:
  - [x] detail view shows the full OVR + market-value trajectory across all seasons (`seasons[].ovr` / `marketValue`) — *API + FE shipped*
  - [x] per-season goals/assists/matches are viewable separately (`seasons[].clubs`) — *API + FE shipped*
- **Data / schema impact**: none — already persisted (`PlayerOvrHistory` unique `playerId+season`; `PlayerSeasonStats` unique `playerId+clubStint+season`; `LoanSpellStats`). Read/aggregate only, no migration.
- **Key files**: `features/players/players.service.ts` (`fetchPlayerById`), `players.routes.ts` (`playerDetailResponse` schema); test in `players.service.test.ts` ("getPlayerById — per-season history (F-004)").
- **FE (shipped)**: the squad detail view consumes `seasons` (OVR/MV chart + per-season table) and `loanSpells` (separate "while on loan" section). Tagged Both in [4.0 Overview](4.0_Overview.md).
- **Related**: [[1.2 Roadmap]] H2 aggregated analytics · #4.2 F-003 · #4.4 B-001 (loan spells) · gated by #4.3 I-006 retaining old data

### F-005 — Competition coverage for all supported clubs/leagues
- **Status**: API implemented (seed reworked); pending run against prod
- **Size**: M (mostly data)
- **Tier**: FREE
- **Problem / motivation**: every club/league is selectable, but their competitions aren't all present — the `Competition` table was under-seeded relative to the in-memory club list. Concretely, only **6 of 24** countries with selectable clubs had any competition seeded; the other 18 (Portugal, Brazil, Argentina, Scotland, USA, Saudi Arabia, …) had none, so a save for one of their clubs tracked no league/cup.
- **Surface (implemented)**: `prisma/seed-competitions.ts` reworked to **derive its country list from `LEAGUE_TO_COUNTRY`** (`clubs.data.ts`) and cover every country with selectable clubs — domestic league + national cup (+ super cup where one exists). 24 countries, 64 domestic + 4 UEFA competitions (68 total, up from 20). A **coverage guard** throws at seed time if any club country lacks competitions. Idempotent (upsert on `Competition.name @unique`); data-only, no migration.
- **Acceptance criteria**:
  - [x] for any selectable club/league, its domestic competitions (league + cup) are available — *seed covers all 24 club countries*
  - [x] coverage can't silently regress on a roster update — *seed guard + `seed-competitions.test.ts` invariant*
  - [ ] seed run against prod (data-only; idempotent upsert per [[prod-db-migration-safety]])
- **Data / schema impact**: `Competition` rows only (no schema change, no migration). No club→competition mapping needed — wiring stays by country via `getCompetitionIdsByCountry`.
- **Decision (2026-06-22)**: continental cups are **Europe-only** for this item. Non-European continental competitions (Libertadores, Sudamericana, AFC CL, …) would require generalizing `EuropeanCup → ContinentalCup` + the rollover dropdown — split into a separate item, out of scope here.
- **Key files**: `prisma/seed-competitions.ts` (rework + guard), `features/competitions/__tests__/seed-competitions.test.ts` (coverage test). `prisma/sql/seed-netherlands-competitions.sql` is now redundant (Netherlands covered by the unified seed).
- **Dependencies**: `clubs.service.ts` club list + `LEAGUE_TO_COUNTRY`.
- **Related**: #3.6.7 Competitions & Clubs
