# 4.2 Features

Backlog of **features** ‚Äî net-new capability the API does not have today: new endpoints, new resources, new integrations, or a meaningfully new behavior on an existing surface. A feature *adds* what isn't there; it does not fix broken behavior (‚Üí [4.1 Corrections](4.1_Corrections.md)), polish existing behavior (‚Üí [4.3 Improvements](4.3_Improvements.md)), or redefine a current domain rule (‚Üí [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md)).

This is a **living list**, populated from analysis. The directional themes live in [1.2 Roadmap](../01_Product/1.2_Roadmap.md); this file is the concrete, item-level backlog those themes resolve into.

## Scope

**Belongs here**
- New endpoints / resources (e.g., aggregated analytics, export/snapshot, bulk operations).
- New integrations or external surfaces (e.g., streaming chat, new MCP tools).
- A new capability layered onto an existing module that a correct client could not do before.

**Does not belong here**
- Restoring intended-but-broken behavior ‚Üí [4.1 Corrections](4.1_Corrections.md).
- Making an existing capability faster / cleaner / better covered ‚Üí [4.3 Improvements](4.3_Improvements.md).
- Changing the meaning of an existing rule or contract ‚Üí [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md).

## Features

### F-001 ‚Äî Server-side chat history persistence
- **Status**: shaped (Roadmap H2)
- **Size**: M
- **Tier**: PRO (chat is PRO)
- **Problem / motivation**: chat history lives in the SPA's `localStorage` ‚Äî no roaming across devices, no continuity, no usage analytics. **Verified**: `features/chat/chat.service.ts` performs no Prisma writes.
- **Proposed surface**: `ChatConversation` / `ChatMessage` tables; endpoints to list/create conversations and append/read messages; `chat.service` persists each turn (user-scoped, optionally save-scoped).
- **Acceptance criteria**:
  - [ ] messages survive reload and device switch
  - [ ] history scoped to the user (and PRO-gated)
- **Data / schema impact**: new tables + migration.
- **Answers U5**: **shortlist and saved searches already persist** ‚Äî `ShortlistItem` and `SavedSearch` Prisma models exist with services. Only chat history is missing, so this item covers chat only.
- **Related**: [[1.2 Roadmap]] H2 ¬∑ pairs with #4.3 I-005 (persona)

### F-002 ‚Äî "Recall" a loaned-out player before the new season
- **Status**: shaped
- **Size**: M
- **Tier**: FREE (the squad/loan hub is free)
- **Problem / motivation**: a loaned-out player can only come back on the next season advance; the user wants to recall mid-season.
- **Proposed surface**: a recall action endpoint that re-attaches the player to the current stint and ends the loan early.
- **Acceptance criteria**:
  - [ ] player returns to the squad immediately
  - [ ] loan record reflects the early return; stats handled per #4.4 B-001
- **Data / schema impact**: possibly loan end metadata on `Transfer`/`Player`.
- **Dependencies**: coordinate with #4.4 B-001 (loan stat attribution) and #4.1 C-001 (aging).
- **Related**: #4.4 B-001, #4.1 C-001, #4.4 B-002

### F-003 ‚Äî History: show the clubs a ranked player played for with you
- **Status**: shaped
- **Size**: S
- **Tier**: FREE
- **Problem / motivation**: the history ranking shows the player but not the clubs (stints) he featured for across your save.
- **Proposed surface**: include the player's club/stint list in the history ranking response.
- **Acceptance criteria**:
  - [ ] each ranked player shows the clubs (stints) he appeared for with you
- **Data / schema impact**: none ‚Äî derivable from `PlayerSeasonStats.clubStintId` ‚Üí `ClubStint.club`.
- **Related**: #4.3 I-004 (history top-N)

### F-004 ‚Äî Full per-season player history in squad detail
- **Status**: shaped (Roadmap H2)
- **Size**: M
- **Tier**: FREE
- **Problem / motivation**: the squad "eye"/detail view shows only the last-season OVR / market-value delta, not the full trajectory or per-season stats.
- **Proposed surface**: an endpoint returning a player's full `PlayerOvrHistory` (OVR + marketValue per season) and `PlayerSeasonStats` (goals/assists/matches/cards per season, per club stint).
- **Acceptance criteria**:
  - [ ] detail view shows the full OVR + market-value trajectory across all seasons
  - [ ] per-season goals/assists/matches are viewable separately
- **Data / schema impact**: none ‚Äî already persisted (`PlayerOvrHistory` unique `playerId+season`; `PlayerSeasonStats` unique `playerId+clubStint+season`). Read/aggregate endpoint only.
- **Related**: [[1.2 Roadmap]] H2 aggregated analytics ¬∑ #4.2 F-003 ¬∑ gated by #4.3 I-006 retaining old data

### F-005 ‚Äî Competition coverage for all supported clubs/leagues
- **Status**: shaped
- **Size**: M (mostly data)
- **Tier**: FREE
- **Problem / motivation**: every club/league is selectable, but their competitions aren't all present ‚Äî the `Competition` table is under-seeded relative to the in-memory club list.
- **Proposed surface**: extend the competitions seed (`db:seed-competitions`) to cover every league/country in the club list; surface them in the selection UI.
- **Acceptance criteria**:
  - [ ] for any selectable club/league, its domestic + relevant continental competitions are available
- **Data / schema impact**: `Competition` rows; possibly a club‚Üícompetition mapping.
- **Dependencies**: `clubs.service.ts` club list + `LEAGUE_TO_COUNTRY`.
