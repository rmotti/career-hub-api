# 4.2 Features

Backlog of **features** — net-new capability the API does not have today: new endpoints, new resources, new integrations, or a meaningfully new behavior on an existing surface. A feature *adds* what isn't there; it does not fix broken behavior (→ [4.1 Corrections](4.1_Corrections.md)), polish existing behavior (→ [4.3 Improvements](4.3_Improvements.md)), or redefine a current domain rule (→ [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md)).

This is a **living list**, populated from analysis. The directional themes live in [1.2 Roadmap](../01_Product/1.2_Roadmap.md); this file is the concrete, item-level backlog those themes resolve into.

## Scope

**Belongs here**
- New endpoints / resources (e.g., aggregated analytics, export/snapshot, bulk operations).
- New integrations or external surfaces (e.g., streaming chat, new MCP tools).
- A new capability layered onto an existing module that a correct client could not do before.

**Does not belong here**
- Restoring intended-but-broken behavior → [4.1 Corrections](4.1_Corrections.md).
- Making an existing capability faster / cleaner / better covered → [4.3 Improvements](4.3_Improvements.md).
- Changing the meaning of an existing rule or contract → [4.4 Business Rule Changes](4.4_Business_Rule_Changes.md).

## Sizing & status legend

- **Size**: `S` (≤1 day) · `M` (a few days) · `L` (a week+) · `XL` (multi-week / needs breakdown).
- **Tier impact**: which plan it touches (FREE hub vs PRO Scout surface — see [2.1 Business Rules](../02_Domain/2.1_Business_Rules.md#identity-plans-sessions)).

Status: `idea` → `shaped` → `accepted` → `in-progress` → `shipped`.

## Features

### F-001 — Server-side chat history persistence
- **Status**: shipped (2026-06-19)
- **Size**: M
- **Tier**: PRO (chat is PRO)
- **Problem / motivation**: chat history lived in the SPA's `localStorage` — no roaming across devices, no continuity, no usage analytics.
- **Surface (implemented)**:
  - `POST /chat/conversations` — create a conversation (`{ title?, saveId? }`); user-scoped, optionally save-scoped.
  - `GET /chat/conversations` — list conversations (`?saveId=` filter); ordered by `updatedAt desc`.
  - `DELETE /chat/conversations/:conversationId` — delete conversation + cascade messages.
  - `GET /chat/conversations/:conversationId/messages` — list messages ordered `createdAt asc`.
  - `POST /chat/messages` — extended with optional `conversationId`: when set, persists the user turn + assistant reply and auto-derives `previousResponseId` from the last assistant message. Backward-compatible: omitting `conversationId` works exactly as before.
- **Acceptance criteria**:
  - [x] messages survive reload and device switch (persisted in `ChatMessage`, read via GET)
  - [x] history scoped to the user (and PRO-gated via existing scope in `app.ts`)
- **Data / schema impact**: `ChatRole` enum + `ChatConversation` + `ChatMessage` models; migration `20260619000002_add_chat_conversations`. `ChatConversation.saveId` FK → `Save` (`SET NULL` on delete). `ChatMessage` FK → `ChatConversation` (`CASCADE`).
- **Key files**: `features/chat/conversations.service.ts` (CRUD + persistence), `chat.controller.ts`, `chat.routes.ts`; `prisma/schema.prisma`.
- **Answers U5**: **shortlist and saved searches already persist** — `ShortlistItem` and `SavedSearch` Prisma models exist with services. Only chat history was missing, now closed.
- **Related**: [[1.2 Roadmap]] H2 · pairs with #4.3 I-005 (persona)

### F-002 — "Recall" a loaned-out player before the new season
- **Status**: done (code complete on main; no migration; pending deploy)
- **Size**: M
- **Tier**: FREE (the squad/loan hub is free)
- **Problem / motivation**: a loaned-out player can only come back on the next season advance; the user wants to recall mid-season.
- **Surface (implemented)**: `POST /saves/:saveId/players/:playerId/recall` — re-attaches the player to the current stint (`status=Role`), ends the loan early, and ensures his `PlayerSeasonStats` for the current season. Audited as `player.loan_recall`.
- **Acceptance criteria**:
  - [x] player returns to the squad immediately
  - [x] does not age on recall (aging happens only on a season advance — decided 2026-06-19)
  - [x] no double-return: the season-advance loan-return pass naturally skips him (no longer `status=Loan` + `activeClubStintId=null`)
  - [x] loan duration honored: `Transfer.returnSeason` now exists (#4.4 B-002); recall ends the loan by flipping the player back to `Role` (the season-advance pass then skips him), so no separate "ended-early" flag is needed
  - [x] informational loan-spell stats preserved on recall — recall never touches `LoanSpellStats` rows (#4.4 B-001)
- **Data / schema impact**: none for F-002 itself (no migration). Loan-end duration ships via #4.4 B-002; loan-spell stats via #4.4 B-001.
- **Dependencies**: builds on #4.1 C-001 (aging, done); coordinates with #4.4 B-001 (loan stats, done) and #4.4 B-002 (2-season, done).
- **Related**: #4.4 B-001, #4.1 C-001, #4.4 B-002

_Implemented (2026-06-19): service `recallLoanedPlayer` in [players.service.ts](../../../src/features/players/players.service.ts), controller + `POST .../recall` route in [players.routes.ts](../../../src/features/players/players.routes.ts). Guards: 400 if the player is not on loan from the current club. Tests in `players.service.test.ts` ("recallLoanedPlayer (F-002)"). No DB migration._

### F-003 — History: show the clubs a ranked player played for with you
- **Status**: shaped
- **Size**: S
- **Tier**: FREE
- **Problem / motivation**: the history ranking shows the player but not the clubs (stints) he featured for across your save.
- **Proposed surface**: include the player's club/stint list in the history ranking response.
- **Acceptance criteria**:

### F-004 — Full per-season player history in squad detail
- **Status**: shaped (Roadmap H2)
- **Size**: M
- **Tier**: FREE
- **Problem / motivation**: the squad "eye"/detail view shows only the last-season OVR / market-value delta, not the full trajectory or per-season stats.
- **Proposed surface**: an endpoint returning a player's full `PlayerOvrHistory` (OVR + marketValue per season) and `PlayerSeasonStats` (goals/assists/matches/cards per season, per club stint).
- **Acceptance criteria**:
  - [ ] detail view shows the full OVR + market-value trajectory across all seasons
  - [ ] per-season goals/assists/matches are viewable separately
- **Data / schema impact**: none — already persisted (`PlayerOvrHistory` unique `playerId+season`; `PlayerSeasonStats` unique `playerId+clubStint+season`). Read/aggregate endpoint only.
- **Related**: [[1.2 Roadmap]] H2 aggregated analytics · #4.2 F-003 · gated by #4.3 I-006 retaining old data
